OO-DB, NoSQL, and Data Lakes

Safety-net primer on the post-relational landscape in case it appears on the final.

Relational databases are the tidy filing cabinets of the data world: every drawer labeled, every folder the same shape. Most business data fits in filing cabinets beautifully. But when the data is a photo, a graph of social connections, a half-gigabyte sensor log, or a document with nested structure, the filing-cabinet metaphor breaks down. OO databases, NoSQL systems, and data lakes are different filing systems for different shapes of information.

This page is a safety-net tour. Week 14 of the course confirmed the final will focus on the five core topics (DB concepts, RA, SQL, ER, normalization); if OO-DB or NoSQL appears, it will be a short definition or conceptual question worth ~5% of the exam at most.

Part A: Object-Oriented Databases

PART A

An OO-DB stores objects directly the way your programming language uses them: with classes, inheritance, and methods. Instead of shredding an Employee object into rows across three relational tables and rebuilding it in application code on every query (the "object-relational impedance mismatch"), the database stores it as a single object graph. If your domain has deeply nested types (CAD drawings, multimedia, scientific data, chemistry compounds), OO-DBs are more natural than relational.

Core OO-DB concepts:

Object
An instance of a class with identity (OID), state (attributes), and behavior (methods).
Class
A template defining the structure and behavior of its objects.
Inheritance
A subclass inherits attributes and methods from a superclass (Manager inherits from Employee).
Encapsulation
Objects expose methods; internal state is hidden behind the interface.
ODL
Object Definition Language: the OO-DB counterpart of SQL's DDL (create table).
OQL
Object Query Language: the OO-DB counterpart of SQL's DML (select).

Why OO-DBs exist. Complex data types (images, graphs, CAD, chemistry), direct mapping to OO program code (no ORM layer), persistent object references without foreign-key joins. Downsides: less mature than relational, weaker query optimizer, vendor lock-in.

Part B: NoSQL Families

PART B

NoSQL ("not only SQL") is an umbrella for four very different database styles that share only one thing: they chose horizontal scaling and flexible schemas over strict ACID guarantees and relational joins. Each family is optimized for a different data shape.

FamilyData ModelExampleBest For
Key-valueString key → opaque blobRedis, DynamoDBcaches, session stores
DocumentJSON-like documentsMongoDB, Couchbasecontent management, catalogs
Column-familySparse wide rows grouped by columnCassandra, HBasetime-series, write-heavy analytics
GraphNodes + edges with propertiesNeo4j, Neptunesocial networks, recommendation, fraud

Key distinguishing features:

Schema-less
Documents in the same collection can have different fields (flexibility over guarantees).
Horizontal scaling
Data is partitioned across many commodity machines; scales out, not just up.
BASE not ACID
Basically Available, Soft state, Eventual consistency. Trade strict consistency for availability and performance (see CAP theorem).
No joins (mostly)
Queries are single-collection; you denormalize data and embed related info inside documents.

Part C: Data Lakes

PART C

A data warehouse is a curated library: books cataloged, organized, sorted before being shelved (ETL-then-store, schema-on-write). A data lake is a storage locker you throw raw crates into; you organize on retrieval (store-then-ETL, schema-on-read). Data lakes accept any format (CSV, JSON, logs, images, Parquet), store it cheaply on object storage (S3, Azure Blob), and let different consumers each impose their own structure at query time.

Data warehouse vs data lake:

DimensionData WarehouseData Lake
SchemaOn write (fixed upfront)On read (applied at query)
Data shapeStructured, cleanedAny format, raw
CostHigher (compute + premium storage)Lower (object storage)
UsersAnalysts, BI toolsData scientists, ML pipelines
QueriesSQL / OLAPSpark, Hive, notebook code
GovernanceStrict (curated)Looser (raw storage)

Data lakehouse. A recent hybrid (Databricks Delta Lake, Iceberg) that adds ACID and schema enforcement on top of data-lake storage, trying to combine the flexibility of a lake with the reliability of a warehouse. Not on this exam, but worth knowing as context.

Why This Matters

The relational model will remain dominant for transactional business data for the foreseeable future because ACID guarantees and SQL's expressiveness are genuinely hard to replicate. But modern systems are almost always polyglot: Postgres for transactions, Redis for caching, Elasticsearch for full-text, S3 for blobs, Neo4j for graph analytics. Knowing when to reach for a non-relational tool is the practical take-away of this chapter.

Glossary

OID
Object Identifier. Immutable unique handle for an object in an OO-DB, analogous to a primary key but opaque.
ODL / OQL
Object Definition Language / Object Query Language. The OO-DB analogs of SQL's DDL / DML.
Impedance mismatch
The friction between an OO program's object model and a relational DB's row-and-table model. OO-DBs remove it; ORMs (Hibernate, Entity Framework) paper over it.
ACID
Atomicity, Consistency, Isolation, Durability. The transaction guarantees of traditional databases.
BASE
Basically Available, Soft state, Eventual consistency. The looser guarantee common in NoSQL.
CAP theorem
A distributed system can give you at most two of: Consistency, Availability, Partition-tolerance. NoSQL systems typically pick AP; relational picks CP.
Schema-on-read
Store data raw; impose structure at query time. Hallmark of data lakes.
Schema-on-write
Enforce structure when data is stored; reject non-conforming inputs. Hallmark of relational DBs and data warehouses.
Polyglot persistence
Using multiple database technologies in one application, each chosen for the data shape it handles best.
