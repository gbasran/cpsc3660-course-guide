The Relational Model

Tuples, attributes, domains, and the integrity rules that turn a spreadsheet into a database.

Part A: Tuples, Attributes, Domains

Physical analogy; a well-disciplined spreadsheet.

Open a spreadsheet, say Excel. The top row is labels: empid, name, office, age. Every row below is one person's record. Each column is promised to hold one kind of thing (ages are numbers, names are strings); that promise is the domain. Two properties make it a relation rather than just a spreadsheet: (1) row order does not matter (sorting by name or by empid gives the same relation), (2) no two rows are completely identical. That second property is a big deal; Excel will happily let you paste the same row five times, but a relation forbids it. Within each row, the cells are atomic; a cell cannot itself contain a list of five phone numbers. If you need that, you make a separate relation.
Definition: relation, tuple, attribute, domain.
Relation
A set of tuples, all sharing the same schema. Denoted $R(A_1, A_2, \ldots, A_n)$ where each $A_i$ is an attribute.
Tuple
A row; one element of the relation. $t = (v_1, v_2, \ldots, v_n)$ where $v_i \in \text{dom}(A_i)$.
Attribute
A column; named component of a tuple. Each has a declared domain.
Domain
The set of legal values for an attribute. Integer, varchar(40), date, boolean, etc.
Degree (arity)
Number of attributes (columns). employee has degree 4.
Cardinality
Number of tuples (rows) currently in the relation.
First Normal Form (1NF)
Every attribute value is atomic; no nested tuples, no sets inside cells. The baseline of the relational model.

Reading a relation carefully

Step 1: Schema and instance

Schema: employee(empid, name, office, age). Degree 4.

Cardinality = 3.

Step 2: Row order does not matter

Swap row 1 and row 3; the relation is the same. SQL's ORDER BY imposes a presentation order on a query result, but the underlying relation has no order.

Step 3: No duplicate rows

Attempting to add another (E01, Ada, D-210, 34) is a no-op at the set level. (Note: SQL tables technically allow duplicates because of the SELECT bag semantics, but the formal relational model does not; this is why we have DISTINCT.)

Mental model: a relation is a SET of tuples.

Part B: Schema vs Instance (revisited)

Physical analogy; blueprint vs house.

We met this split on Page 01. Here we use it in the pure relational sense. The schema is the declaration: "this relation is called employee, has four columns, their domains are X/Y/Z, its primary key is empid." The instance is whatever set of tuples currently satisfies that declaration. A query like select * from employee returns the current instance; the schema is the thing you declared once with CREATE TABLE.
Notation.
$R(A_1, A_2, \ldots, A_n)$
Schema of relation $R$ with attributes $A_1$ through $A_n$. Written once.
$r(R)$ or $r$
An instance; a specific set of tuples valid under schema $R$. Varies with time.
$t \in r$
Tuple $t$ is currently in the instance.
$t[A]$
The value of attribute $A$ in tuple $t$. So $t[\text{name}]$ might be "Ada."

Part C: Primary Keys

Physical analogy; the unique barcode on every library book.

Every book in the university library has a barcode sticker. No two books share a barcode, and every book has one (never blank). The librarian uses the barcode as the identifier when checking books in and out; it does not matter if two books share a title or an author. The primary key is exactly that sticker; one designated attribute (or tuple of attributes) that the DBMS treats as the canonical row identifier. It enforces two promises: uniqueness (no duplicates across the instance) and non-nullness (every row has a value).
Rule: Primary key enforcement.

For a relation $r$ with primary key $K$:

(1) Uniqueness: $\forall t_1, t_2 \in r,\ t_1[K] = t_2[K] \Rightarrow t_1 = t_2$.
(2) Entity integrity: $\forall t \in r,\ t[K] \neq \text{NULL}$.

The DBMS rejects any INSERT or UPDATE that would violate either rule.

Declaring a composite primary key

Step 1: Identify the uniqueness

In borrow(empid, isbn, borrow_date), neither empid nor isbn alone is unique (one employee borrows many books; one book is borrowed by many employees). The pair (empid, isbn) uniquely identifies the borrow record (assuming you borrow each book at most once).

Step 2: Declare in SQL
create table borrow (
  empid char(5),
  isbn char(13),
  borrow_date date not null,
  primary key (empid, isbn)
);

Composite PK (empid, isbn). Both columns are automatically NOT NULL; the pair must be unique.

Part D: Foreign Keys and Referential Integrity

Physical analogy; a coat-check ticket.

At a restaurant, you hand your coat to the attendant and she gives you a numbered paper ticket. The ticket itself is useless without the coat; it is a reference. When you come back and hand over ticket #42, she walks to the rack and retrieves the coat hanging from hook #42. The system only works because the tickets in circulation match actual coats on hooks; no stray tickets, no empty hooks with tickets out. A foreign key is a coat-check ticket pointing to a row in another table; referential integrity is the restaurant's rule that every ticket in your pocket must correspond to a real coat.
Definition: Foreign key and referential integrity.
Foreign key
A set of attributes $F$ in relation $R_1$ that references the primary key $K$ of relation $R_2$. Values in $F$ must either be NULL (if allowed) or appear in some tuple's $K$ in $R_2$.
Referential integrity
$\forall t_1 \in r_1,\ t_1[F] = \text{NULL}\ \lor\ \exists t_2 \in r_2 \text{ such that } t_2[K] = t_1[F]$.
Referencing relation
The one holding the FK. Contains the "ticket."
Referenced relation
The one whose PK is being pointed at. Contains the "coat."

What FKs enforce, in tuples

Step 1: Setup

employee = {(E01, Ada), (E02, Boris)}.
borrow = {(E01, 978-A, 2026-03-01)}; the FK borrow.empid → employee.empid is satisfied because E01 exists.

Step 2: Attempt an insert that violates
insert into borrow values ('E99', '978-A', '2026-04-17');

E99 is not in employee. The DBMS rejects the insert with "foreign key violation."

Step 3: Attempt to update the referenced row
update employee set empid='E10' where empid='E01';

This would leave borrow's row pointing at a deleted-and-renumbered E01. Depending on the ON UPDATE rule (CASCADE, RESTRICT, SET NULL), the DBMS either propagates the change to borrow, blocks the update, or nulls out the FK.

These cascade rules are on Page 01, Part F.

Exam Pattern

The prof has asked: "what is the difference between a primary key and a foreign key?" One-sentence answer: a PK identifies a row within its own relation; a FK points from one relation to another's PK. Expand with: PK enforces uniqueness and non-null; FK enforces referential integrity (every value must exist in the target PK).

Part E: Null Values and NOT NULL

Physical analogy; the blank line on a form.

Paper forms have three possible states per field: "filled in," "explicitly marked N/A," and "just left blank." Databases treat all three as NULL, and NULL means "value unknown or inapplicable." It is not zero, not an empty string, not false. You cannot test NULL with = because NULL is not a value you can compare to; age = NULL is itself NULL, which is treated as false in a WHERE clause. To test for it you use the keyword IS NULL (or IS NOT NULL). Any column declared NOT NULL refuses any row that leaves it blank; primary keys are automatically NOT NULL; foreign keys may or may not be.
Rules for NULL.
Comparison
x = NULL evaluates to NULL, not true/false. Always use IS NULL or IS NOT NULL.
Arithmetic
Any arithmetic with NULL yields NULL. 5 + NULL = NULL, NULL * 0 = NULL.
Three-valued logic
WHERE clauses evaluate to true, false, or unknown. Only true rows pass. unknown is filtered out.
Aggregates
COUNT(*) counts all rows; COUNT(column) ignores NULLs; SUM, AVG, MIN, MAX all ignore NULLs.
NOT NULL constraint
A column-level declaration. The DBMS rejects any insert/update that would leave the column NULL.

The NULL gotcha

Step 1: Data
empidnameoffice
E01AdaD-210
E02BorisNULL
Step 2: The wrong query
select name from employee where office = NULL;

Returns zero rows. The comparison office = NULL evaluates to NULL for every row; no row passes.

Step 3: The right query
select name from employee where office is null;

Returns Boris. Always use IS NULL / IS NOT NULL.

Why This Matters

Nearly every SQL exam question has a NULL trap hidden in it. If the question says "find employees whose office is not D-210," the naive query WHERE office <> 'D-210' misses rows where office IS NULL, because NULL <> 'D-210' is NULL, not true. The safe version is WHERE office IS NULL OR office <> 'D-210'. The prof has tested this.

Practical habit. When you design a schema, declare NOT NULL on every column except ones that genuinely allow "unknown" as a state. This eliminates a huge class of query bugs. If a column must accept missing values, write SQL with that in mind from the start.

Glossary

Relation
A set of tuples over a shared schema; the relational model's only structure.
Tuple
One row; element of a relation.
Attribute
A named column of a relation, with a declared domain.
Domain
The allowed values for an attribute (integer, varchar, date, etc.).
Degree (arity)
Number of attributes.
Cardinality
Number of tuples in the current instance.
1NF
First Normal Form; all attribute values atomic.
Primary key (PK)
Designated unique identifier; NOT NULL, UNIQUE.
Foreign key (FK)
Attribute(s) in one relation referencing a PK in another.
Referential integrity
FK values must be NULL or match an existing PK.
Entity integrity
PK values must never be NULL.
NULL
Marker for "unknown" or "inapplicable"; propagates through arithmetic and comparisons.
NOT NULL
Column-level constraint forbidding NULL values.
Three-valued logic
Boolean logic extended with "unknown"; WHERE keeps only "true" rows.
