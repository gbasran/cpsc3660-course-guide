10 SQL Problems, Walked Step-by-Step

Built on the three recycled schemas the prof reaches for on every exam: Library, Employee-Works-Company, Bank.

Physical analogy; following a chef through ten plating walkthroughs.

A cooking class does not just give you the recipe; the chef stands next to you, hand on yours, walking through every motion. "Pick up the fish with this grip. Now slide the spatula under from the left, never the right." SQL problems have this same teachable muscle memory. These 10 walkthroughs step through the problem statement, the logic, the query, and the common traps, using the exact schemas the prof will put in front of you on the final.
Schemas used throughout this page.
Library: employee(empid, name, office, age), books(isbn, title, authors, publisher), borrow(empid, isbn, borrow_date).
Bank: branch(branch_name, branch_city, assets), customer(customer_name, customer_street, customer_city), account(account_number, branch_name, balance), depositor(customer_name, account_number), loan(loan_number, branch_name, amount), borrower(customer_name, loan_number).
Employee-Works-Company: employee(employee_name, employee_street, employee_city), works(employee_name, company_name, employee_salary), company(company_name, company_city), manages(employee_name, manager_name).

Problem 1: Simple SELECT + WHERE (Library)

Problem (2 marks; easy). Find the names of all employees over age 30.
Model SQL
select name
from employee
where age > 30;

Walkthrough

Step 1: Identify the relation

Only employee is needed; no joins. Name and age are both in employee.

Step 2: Filter, then project

where age > 30 filters rows; select name keeps only the one column asked for.

Common error

Do not write select *. The question asks for names; projecting everything loses marks for imprecision.

Mirrors A1 Q6(a); A2 Q1(a); Midterm Q8(a)

Problem 2: Multi-Table Join (Library)

Problem (3 marks; medium). Find the names of employees who have borrowed a book titled "Harry Potter".
Model SQL
select distinct e.name
from employee e, borrow b, books k
where e.empid = b.empid
  and b.isbn = k.isbn
  and k.title = 'Harry Potter';

Walkthrough

Step 1: Trace the join chain

Employee ↔ borrow via empid; borrow ↔ books via isbn. Three-table join.

Step 2: Apply the book-level filter

k.title = 'Harry Potter' narrows the books table to just the target book.

Step 3: Project distinct names

Without distinct, an employee who borrowed the book twice would appear twice.

Common error

Forgetting distinct. Forgetting one of the two join predicates produces a Cartesian-product explosion.

Mirrors A1 Q7(a); A2 Q1(a)

Problem 3: GROUP BY + Aggregate (Library)

Problem (3 marks; medium). For each publisher, count the number of books.
Model SQL
select publisher, count(*) as book_count
from books
group by publisher;

Walkthrough

Step 1: Group by the non-aggregate attribute

The "for each" phrasing is the GROUP BY signal. Group by publisher; one row per distinct publisher.

Step 2: Aggregate over the group

count(*) counts rows within each group. Alias it for readability.

Common error

Writing select title, count(*); non-aggregate columns that are not in the GROUP BY cannot appear in SELECT.

Mirrors A2 Q2(b); Midterm Q8(b)

Problem 4: GROUP BY + HAVING + Threshold (Bank)

Problem (3 marks; medium; 95% guaranteed on the final). Find the branches where the average account balance is greater than 10,000.
Model SQL
select branch_name, avg(balance) as avg_bal
from account
group by branch_name
having avg(balance) > 10000;

Walkthrough

Step 1: GROUP BY branch

Each branch is one row in the output; aggregate balances within the group.

Step 2: Filter groups with HAVING, not WHERE

WHERE filters rows before grouping; HAVING filters groups after aggregation. Averages live at the group level, so the threshold lives in HAVING.

Common error

Writing where avg(balance) > 10000; SQL does not allow aggregate functions in WHERE. The error is fatal.

Mirrors A1 Q5; A2 Q4; A3 Q1(c); Midterm Q5(b). The guaranteed pattern.

Problem 5: Correlated Subquery (Employee-Works)

Problem (4 marks; hard). Find the names of employees who earn more than the average salary at their own company.
Model SQL
select w.employee_name
from works w
where w.employee_salary > (
  select avg(w2.employee_salary)
  from works w2
  where w2.company_name = w.company_name
);

Walkthrough

Step 1: Recognize the correlation

"Average at their own company" changes per employee; the subquery must depend on the outer row. Correlated, not independent.

Step 2: Alias both copies of works

The outer reference is w; the inner is w2. The correlation is w2.company_name = w.company_name.

Step 3: Do not over-join

The company table is not needed; works already stores the company name. Adding a join to company is over-joining and wastes marks.

Common error

Writing an uncorrelated subquery (avg(employee_salary) over all of works). That compares to the global average, not the per-company average.

Mirrors A3 Q3(a) verbatim

Problem 6: NOT EXISTS for Universal Quantification (Library)

Problem (4 marks; hard). Find the names of employees who have borrowed every book published by "Springer".
Model SQL
select e.name
from employee e
where not exists (
  select 1
  from books k
  where k.publisher = 'Springer'
  and not exists (
    select 1
    from borrow b
    where b.empid = e.empid
    and b.isbn = k.isbn
  )
);

Walkthrough

Step 1: Translate "every" into double negation

"Borrowed every Springer book" becomes "there does not exist a Springer book that this employee has not borrowed." Two negations.

Step 2: Outer filter on employees

The employee $e$ is kept iff the outer NOT EXISTS is true (no Springer book is unborrowed by $e$).

Step 3: Inner NOT EXISTS checks "has not borrowed"

For a fixed Springer book $k$, the inner NOT EXISTS says "no borrow row connects this employee to this book".

Common error

Trying to use = all or count(...) = (select count(...) ...). Correct but clunky; the prof strongly prefers NOT EXISTS, which mirrors RA division directly.

Mirrors A2 Q2(c); A3 Q3(b); the SQL side of RA division

Problem 7: Subquery with IN / NOT IN (Bank)

Problem (3 marks; medium). Find the names of customers who have an account but no loan.
Model SQL
select distinct customer_name
from depositor
where customer_name not in (
  select customer_name
  from borrower
);

Walkthrough

Step 1: Identify the two relations

depositor records account holders; borrower records loan holders. "Account but no loan" is depositor - borrower on customer_name.

Step 2: Express as NOT IN

A customer is in the answer iff they appear in depositor and do NOT appear in borrower.

Alternative

Using EXCEPT: (select customer_name from depositor) except (select customer_name from borrower);. Functionally equivalent.

Mirrors A3 Q2

Problem 8: View Definition (Bank)

Problem (3 marks; medium). Create a view $savings\_customers\_no\_loan$ listing the names of customers who have a savings account (branch "Brighton") but no loan at any branch.
Model SQL
create view savings_customers_no_loan as
select distinct d.customer_name
from depositor d, account a
where d.account_number = a.account_number
  and a.branch_name = 'Brighton'
  and d.customer_name not in (
    select customer_name from borrower
  );

Walkthrough

Step 1: Wrap a SELECT with CREATE VIEW

Views are named queries; the syntax is create view <name> as <select-query>.

Step 2: Join depositor to account to filter by branch

Depositor only names customers; the branch lives in account.

Step 3: Exclude borrowers via NOT IN

Same trick as Problem 7, embedded inside the view definition.

Mirrors A3 Q1(a) verbatim

Problem 9: INTERSECT / EXCEPT (Bank)

Problem (3 marks; medium). Two parts.
(a) Find customers who have both an account and a loan.
(b) Find customers who have an account but no loan.
Part (a): INTERSECT
(select customer_name from depositor)
intersect
(select customer_name from borrower);
Part (b): EXCEPT
(select customer_name from depositor)
except
(select customer_name from borrower);

Walkthrough

Step 1: Set operators require compatible select lists

Both sides of INTERSECT/EXCEPT must project the same number and type of columns. Here both project one text column customer_name.

Step 2: Duplicates

INTERSECT and EXCEPT remove duplicates by default in SQL (unlike UNION ALL). If you need duplicates preserved, use intersect all / except all.

Common error

Mixing tables that have different schemas on each side without explicit projection; always project explicitly.

Mirrors A3 Q2; classic Bank-schema test of set ops

Problem 10: Complex Correlated Subquery (Employee-Works)

Problem (5 marks; hard). Find the names of all companies located in every city where "IBM" is located.
Model SQL (NOT EXISTS form)
select distinct c.company_name
from company c
where not exists (
  select 1
  from company ibm
  where ibm.company_name = 'IBM'
  and not exists (
    select 1
    from company c2
    where c2.company_name = c.company_name
    and c2.company_city = ibm.company_city
  )
);

Walkthrough

Step 1: Recognize universal quantification again

"In every city where IBM is" = "there is no IBM city where this company is NOT".

Step 2: Structure the three levels

Outer: candidate company $c$. Middle: for each IBM city, check. Inner: does $c$ have a row in that same city?

Step 3: Note the multi-row assumption

Note: in this schema company(company_name, company_city), a company can have multiple rows (one per city). That is what makes "every city" meaningful. IBM has rows for every city it operates in.

Common error

Missing the IBM alias and writing a single where company_name = 'IBM'; you lose the ability to iterate over IBM's cities.

Mirrors A3 Q3(b); canonical hard-tier SQL problem

The three-pass drill

Pass 1: read only the problem statement, draft the query on paper from memory. Pass 2: compare your draft to the model answer; note any structural differences. Pass 3: rewrite the query once from scratch without looking. Do this for Problems 4, 5, 6, and 10 (the high-yield and hard ones); those are the discriminators that decide your grade.

Glossary

GROUP BY
Partitions rows by the listed attributes; each partition becomes one output row.
HAVING
Filters groups after aggregation; required when the threshold is on an aggregate function.
Correlated subquery
A subquery that references an outer row; runs once per outer row.
NOT EXISTS
True when the subquery returns zero rows; the SQL form of universal quantification when nested twice.
INTERSECT / EXCEPT
Set operators; intersect returns common tuples, except returns tuples in the left not in the right.
View
A named SELECT query; not materialized by default.
Over-join
Adding a table that is not needed; costs marks for imprecision.
Distinct
Removes duplicate rows from the result; needed when join can produce repeats.
