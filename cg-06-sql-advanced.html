SQL: Advanced

Aggregation, GROUP BY / HAVING, views, subqueries, set operations, and NULL handling; everything beyond the SELECT/FROM/WHERE skeleton.

Part A: Aggregate Functions

Physical analogy; a cashier totaling a sales receipt.

After a busy day the cashier runs the register's "Z-report." It spits out: total number of transactions (count), total dollars (sum), average transaction size (avg), smallest sale (min), largest (max). Each number summarizes an entire day of rows into one scalar. SQL's aggregate functions do the same; you feed them a column (or the special *) and they collapse the whole result set into one value, unless GROUP BY tells them to produce one value per group.
The five aggregates.
count(*)
Number of rows, including rows with all NULLs.
count(col)
Number of rows where col is NOT NULL.
count(distinct col)
Number of distinct non-NULL values in col.
sum(col), avg(col)
Sum and average; NULL values ignored; avg is sum / count(col).
min(col), max(col)
Smallest and largest non-NULL value. Works for numbers, dates, strings.

Simple aggregate queries on employee

select count(*) as num_employees,
      avg(age) as avg_age,
      min(age) as youngest,
      max(age) as oldest
from employee;

Returns ONE row. No GROUP BY was needed because the whole table is treated as a single group.

Part B: GROUP BY

Physical analogy; sorting mail into bins by zip code.

Imagine a bag of mixed mail hitting a sorting table. The sorter partitions it by zip code; at the end there are many little stacks, one per zip. Now a per-stack summary becomes natural: "how many envelopes in zip 90210?" or "heaviest package in zip 60601?" GROUP BY is that partitioning. Once tuples are in their groups, every aggregate function runs per group, producing one row of output per distinct group.
GROUP BY skeleton.
select <group-cols>, <aggregate(...)>
from <tables>
where <row-filter>
group by <group-cols>
having <group-filter>
order by <col>;
SELECT constraint
Any non-aggregate column in SELECT must appear in GROUP BY. Otherwise the query is ambiguous (which value of that column for each group?).
WHERE vs HAVING
WHERE filters rows before grouping. HAVING filters groups after aggregation. Both are allowed in the same query.

Books borrowed per publisher

Step 1: Join borrow and books
select bk.publisher, count(*) as n
from borrow br
join books bk on br.isbn = bk.isbn
group by bk.publisher;
Step 2: Trace it

Each tuple of the join carries one publisher. Partitioning by publisher gives one group per publisher. count(*) counts the rows in each group. Result: one row per publisher with the number of borrow events attributed to it.

Adding order by n desc ranks publishers by popularity.

Part C: HAVING; the "avg > threshold" Pattern

Physical analogy; the sorter sets aside only the heavy stacks.

After sorting mail into zip-code stacks, the sorter flags only the stacks that weigh more than 5 kg. She is not filtering individual envelopes (those stayed in stacks); she is filtering whole stacks by a property of the stack as a group. HAVING is that post-grouping filter. It sees the output of GROUP BY plus aggregates, and lets through only the groups whose aggregate value satisfies the condition.
Rule: when to use HAVING.

If the condition depends on an aggregate (count, sum, avg, min, max over the group), use HAVING. If the condition depends on an individual-row attribute, use WHERE.

Publishers with average book-age over 10 years

Assume books also have a year_published column. Using year 2026:

select publisher, avg(2026 - year_published) as avg_age
from books
group by publisher
having avg(2026 - year_published) > 10;

WHERE would have filtered individual books; HAVING filters whole publisher groups by their average.

Exam Pattern (Midterm Q6(d), A2 Q7(b)) — 95%-guaranteed

The prof tests some variant of "groups where avg(x) > threshold" almost every term. Template:

select group_col, avg(x)
from table
group by group_col
having avg(x) > T;

Variants swap avg for count (branches with more than 3 accounts), sum (publishers whose total-pages-borrowed exceeds N), or min (courses whose lowest grade is still an A). The skeleton is always group-by then having-aggregate-condition; memorize it cold.

Part D: Views

Physical analogy; a saved search shortcut.

You keep running the same three-table join to see "employees and the books they borrowed this month." Rather than retype that query every time, you save it under a name, employee_borrows. Now anywhere you would write that big query, you write select * from employee_borrows. That is a view: a named query that behaves like a virtual table. There are two execution modes: computed on demand (every use re-runs the underlying query against current data) and materialized (the result is stored physically and refreshed periodically; faster to read, stale between refreshes).
Syntax: CREATE VIEW.
create view <name> [(col1, col2, ...)] as
  select ...;

-- use it like a table:
select * from <name> where ...;
Computed (virtual)
Default. The view holds no data; every query expands the view's SELECT inline.
Materialized
create materialized view .... The view's result is stored on disk. Must be refreshed (refresh materialized view) to see new data.
Updatability
Simple single-table views may allow INSERT/UPDATE/DELETE that propagates to the base table. Views with JOINs, GROUP BY, or aggregates are usually read-only.

View for "who borrowed what"

create view who_borrowed as
  select e.empid, e.name, bk.title, br.borrow_date
  from employee e
  join borrow br on e.empid = br.empid
  join books bk on br.isbn = bk.isbn;

select name, title
from who_borrowed
where borrow_date >= '2026-01-01';

The outer query reads as if who_borrowed were a physical table; the DBMS expands it inline.

Part E: Subqueries (Correlated and Uncorrelated)

Physical analogy; asking a question whose answer depends on another question.

An uncorrelated subquery is "find the list of book ISBNs published by Penguin" computed once and then used as a lookup list in the outer query: "give me every borrow whose ISBN is in that list." The inner answer is computed one time, regardless of how many outer rows we examine. A correlated subquery is different: the inner query references a column from the outer row, so it must be re-evaluated for each outer row. "For this specific employee, what is the average salary of people at their company?" The inner computation depends on who the outer row is.
Two shapes of subquery.
Uncorrelated
Inner SELECT can be computed independently. Runs once. Results used via IN, =, <, <=, etc.
Correlated
Inner SELECT references a column from the outer query's table. Logically re-runs once per outer row.
EXISTS / NOT EXISTS
Boolean test: true if the subquery returns at least one row. Commonly used with correlated subqueries.
ALL / ANY / SOME
Quantified comparisons: x > ALL(...) means x exceeds every value; x > ANY(...) means x exceeds at least one.

Uncorrelated: employees who borrowed any Penguin book

select name
from employee
where empid in (
  select br.empid
  from borrow br
  join books bk on br.isbn = bk.isbn
  where bk.publisher = 'Penguin'
);

The inner query runs once; it returns a fixed list of empids.

Correlated: employees earning more than the average at their own company (A3 Q3 style)

Schema: works(employee_name, company_name, employee_salary). Find employees whose salary exceeds the average salary at their own company.

Step 1: Think about what "average at their own company" means

The average depends on which employee we are looking at; two employees at different companies have different reference averages. That is the signature of a correlated subquery.

Step 2: Write the outer query with an alias
select w1.employee_name
from works w1
where w1.employee_salary > (
  select avg(w2.employee_salary)
  from works w2
  where w2.company_name = w1.company_name
);
Step 3: Trace it

For each outer row w1: the DBMS conceptually re-runs the inner query, restricting w2 to the same company as w1; computes the average salary at that company; compares w1's salary against it; keeps the row iff strictly greater.

The key linkage is the predicate w2.company_name = w1.company_name inside the inner WHERE; remove that and the subquery degenerates to "company-wide average" which is wrong.

Exam Pattern (A3 Q3(a))

Any time the English spec uses the possessive ("their own company," "the branch where they work," "the course they are enrolled in") you are likely writing a correlated subquery. Template:

select col
from T1 t1
where t1.x OP (
  select AGG(t2.y)
  from T1 t2    -- same table, or a joined table
  where t2.linking_col = t1.linking_col
);

The linking predicate is non-negotiable. Always use explicit aliases (t1, t2) so the marker can see which reference is outer and which is inner.

Part F: EXISTS, NOT EXISTS, and Division in SQL

Physical analogy; checking whether a drawer contains anything.

EXISTS is a Boolean: "does the drawer contain at least one item?" You do not care what the items are, just whether any exist. NOT EXISTS is the opposite: "is the drawer empty?" In SQL these are used with subqueries: EXISTS (SELECT ...) returns true if the inner query would return at least one row, false otherwise. NOT EXISTS is the standard way to express the "ALL" pattern from relational algebra division in SQL.
Division in SQL via double NOT EXISTS.

"Find employees who have borrowed ALL books from Penguin" translates to "find employees for whom there does NOT EXIST a Penguin book that they have NOT borrowed":

select e.empid, e.name
from employee e
where not exists (
  select *
  from books bk
  where bk.publisher = 'Penguin'
    and not exists (
      select *
      from borrow br
      where br.empid = e.empid and br.isbn = bk.isbn
    )
);

Read aloud: "there is no Penguin book that this employee has not borrowed."

Exam Pattern

When an English problem uses "all" or "every," consider two SQL approaches: (1) the double NOT EXISTS pattern above, or (2) GROUP BY with a COUNT match (count(distinct borrowed_isbn) = (select count(*) from ...penguin...)). Both work; NOT EXISTS is the textbook answer and always correct.

Part G: ALL, ANY, SOME

Quantified comparisons.
x > ALL (subquery)
True iff x is greater than every value the subquery returns. Equivalent to "x > max."
x > ANY (subquery)
True iff x exceeds at least one value. Equivalent to "x > min."
ANY vs SOME
Synonyms. = ANY is equivalent to IN; <> ALL is equivalent to NOT IN (but safer around NULLs).

Employees older than every D-214 employee

select name
from employee
where age > all (
  select age
  from employee
  where office = 'D-214'
);

Equivalent and often cleaner: where age > (select max(age) from employee where office = 'D-214').

Part H: UNION, INTERSECT, EXCEPT

Physical analogy; combining two guest lists.

Two guest lists, same columns (name, rsvp). UNION merges them (everyone on either list, no duplicates). INTERSECT keeps only people on both lists (confirmed double-invited). EXCEPT keeps the people on the first list who are NOT on the second. These are exactly the set operations from relational algebra, surfaced directly as SQL keywords.
Syntax.
(select ... from ...) union (select ... from ...);
(select ...) intersect (select ...);
(select ...) except (select ...);

-- keep duplicates with "all":
(select ...) union all (select ...);
Schema rule
Both SELECTs must produce the same number of columns with compatible types. Column names from the first SELECT are used.
UNION vs UNION ALL
UNION removes duplicates (expensive). UNION ALL keeps duplicates (cheaper). Default is UNION.
EXCEPT
Some DBMSs use MINUS (Oracle). Same meaning.

Customers with an account or a loan (Bank schema)

(select customer_name from depositor)
union
(select customer_name from borrower);

Intersect gives customers with both; except gives customers with an account but no loan.

Part I: NULL Handling Revisited

NULL rules in aggregates and comparisons.
count(*)
Counts all rows, including rows with all-NULL columns.
count(col), sum, avg, min, max
Skip NULL values in col.
IS NULL / IS NOT NULL
Only way to test for NULL. Never = NULL.
NULL propagation
NULL + 5 = NULL, NULL * 0 = NULL, NULL = NULL is NULL (not true).
NOT IN gotcha
If the subquery list contains a NULL, x NOT IN (list) always returns NULL (not true). Use NOT EXISTS instead for safety.

Employees with no office assignment

select name from employee where office is null;

Writing where office = null returns zero rows; the predicate evaluates to NULL for every row.

Why This Matters

A surprising number of exam-losing bugs are NULL handling: using = instead of IS, forgetting that NOT IN fails with NULL-containing lists, forgetting that aggregates skip NULL. Build a mental habit: any column that could be NULL needs explicit handling. When in doubt, use NOT EXISTS instead of NOT IN.

Glossary

COUNT(*)
Number of rows including all-NULL rows.
COUNT(col)
Number of non-NULL values in col.
COUNT(DISTINCT col)
Number of distinct non-NULL values.
SUM / AVG / MIN / MAX
Aggregate functions; all skip NULLs.
GROUP BY
Partition rows by grouping columns; aggregates apply per group.
HAVING
Post-grouping filter; condition on aggregate or grouping column.
View
Named query behaving like a virtual table. Virtual (default) or materialized.
Materialized view
Results stored on disk; must be refreshed to reflect changes.
Uncorrelated subquery
Inner query independent of outer row; runs once.
Correlated subquery
Inner query references an outer column; re-evaluated per outer row.
EXISTS
True if subquery returns at least one row.
NOT EXISTS
True if subquery returns no rows; used for division ("all" pattern).
ALL / ANY / SOME
Quantified comparisons with a subquery-returned set.
UNION / INTERSECT / EXCEPT
Set operations on SELECT results; schemas must match.
UNION ALL
Union that keeps duplicates.
IS NULL / IS NOT NULL
The only correct NULL comparison.
NOT IN gotcha
Returns NULL (not true) if list contains NULL; prefer NOT EXISTS.
