PAGE 18; EXAM PREP

Paper Exam Techniques

How to physically write RA, SQL, ER, and FD work on paper. Symbols by hand, layout conventions, and a 180-minute time budget.

Physical analogy; a surgeon laying out instruments before the cut.

A surgeon does not start the operation and then reach blindly for tools; the tray is laid out in a fixed order before the patient even comes in. The paper exam is the same operating table; the margin, the symbol conventions, the clause layout, the FD-proof numbering, all have to be decided before you sit down. When the clock starts, you should be writing answers, not deciding where to put them.

Part A: Drawing RA Symbols by Hand

The six symbols you will write most often.
$\sigma$ (sigma)
Cursive lowercase sigma. Draw like a curled-up backwards "e" with a small tail trailing down-right. The tail is what distinguishes it from an "o".
$\pi$ (pi)
Two vertical strokes under a flat horizontal roof; like a tiny capital-Pi but lowercase. Keep both legs the same length.
$\bowtie$ (bowtie / natural join)
Two triangles meeting point-to-point, like a bow tie. Draw the left triangle first (point right), then the right triangle (point left), touching at a single vertex in the middle. Do not leave a gap.
$\div$ (division)
The regular arithmetic division sign; a horizontal line with one dot above and one dot below.
$\rho$ (rho)
Cursive lowercase rho; a small "p" with a curl at the top that opens to the left. Sometimes students confuse it with "p"; add the curl distinctly.
$\gamma$ (gamma)
Lowercase gamma; like a lowercase "y" whose tail does not loop. Two slanted strokes meeting near the top, the right leg descending below the line.
Set operators. $\cup$ (union) is a lowercase "u" with rounded bottom. $\cap$ (intersection) is $\cup$ flipped upside-down. $-$ (difference) is a plain minus sign. $\times$ (Cartesian product) is a multiplication sign; do not confuse with "x" letter.
Ex A(i): "names of employees who borrowed a Springer book" // project after select+join

πname(σpublisher='Springer'(
employee ⋈ borrow ⋈ books))

Ex A(ii): "employees who borrowed every Springer book" // "every" → division

πname(employee ⋈
(πempid, isbn(borrow) ÷
πisbn(σpub='Springer'(books))))

Ex A(iii): self-join with rename // rho first, then sigma on cross

R = ρA1(account), S = ρA2(account)
σR.bal > S.bal(R × S)

Writing subscripts cleanly

The rule

Write the operator at full size; drop the subscript to about 60% size and push it slightly below the baseline. Do not shrink the operator to fit the subscript; shrink the subscript to fit under the operator.

Example; selection with a compound predicate
      σ                           <-- full-size
       age > 30 AND name='Ada'    <-- small, below baseline

      (employee)                  <-- argument in parens

    π
     name, age

    (employee ⋈ borrow)
Common error

Writing the subscript the same size as the operator, then running out of room on the line. Leave generous vertical spacing between rows; 1.5x line height is not too much.

Part B: SQL Layout on Paper

Layout rules.
  1. Lowercase keywords; matches the prof's style, and lowercase letters are faster to write.
  2. One clause per line; select, from, where, group by, having, order by. Each starts at the left margin.
  3. Indent subqueries one level (about 2 characters).
  4. Align AS aliases in a column when you have several.
  5. Explicit aliases (E1, E2, etc.) for every self-join.
Canonical paper SQL layout
select e.name, avg(w.employee_salary) as avg_sal
from employee e, works w
where e.employee_name = w.employee_name
  and w.company_name in (
    select company_name
    from company
    where company_city = 'Lethbridge'
  )
group by e.name
having avg(w.employee_salary) > 50000;
Ex B: handwritten-shape SQL // lowercase kw; one clause per line

select e.name, avg(w.salary) as avg_sal
from employee e, works w
where e.emp_name = w.emp_name
and w.company in (
select company
from company
where city = 'Lethbridge'
)
group by e.name
having avg(w.salary) > 50000 ;
↑ each and lines up under the first predicate. Subquery indents one level. Final ; visible.

What marks get lost when the layout is sloppy

Compressed into one line

The grader misses your WHERE joining predicates; easy to undercount. Lose 0.5-1 marks.

Mixed-case keywords

Some lowercase, some uppercase. Visually inconsistent; not wrong, but suggests rushed work.

Missing subquery indent

Hard to see where the subquery ends; grader may mistake the wrong clause for an outer clause. Clarity lost; marks lost.

Part C: ER Notation Conventions

Shape grammar.
Rectangle
Entity. Single border for strong entity; double border for weak entity.
Oval
Attribute. Single border for simple. Double border for multi-valued. Dashed border for derived. Branching sub-ovals for composite.
Diamond
Relationship. Single border for normal; double border for identifying relationship (the one that identifies a weak entity).
Underline
Primary key attributes; solid underline for strong entity's PK; dashed underline for weak entity's discriminator.
Edge label
Cardinality. Write 1, N, M on each edge; or use arrowheads (single arrow = "one"; double arrow = "many").
ISA triangle
Triangle pointing up from subtype to supertype; the word "ISA" written inside or next to it.
Double line
Total participation; every entity instance must participate in the relationship.

A mini ER drawn with ASCII (use real shapes on paper)

           (name)   (age)          (isbn)   (title)
              \     /                \      /
               \   /                  \    /
             +----+                +-------+
             |Emp |                | Books |
             +----+                +-------+
                \                       /
                 \    1          M     /
                  \____<>_borrow<>____/
                       N          1
                           |
                        (date)     <-- attribute on relationship

Primary keys: empid on Emp (underlined), isbn on Books (underlined). Cardinality 1:N between Emp and borrow (each employee has N borrows; each borrow has 1 employee). Here we model borrow as a relationship with attribute date.

Paper order. Pencil first; place entities at the top and spread them out. Draw relationships (diamonds) between them. Add attributes last. Once everything fits, ink over in pen. Never start in pen.

Part D: FD Proof Layout

The rules (Armstrong's axioms plus derived rules).
Reflexivity
If $\beta \subseteq \alpha$, then $\alpha \to \beta$.
Augmentation
If $\alpha \to \beta$, then $\alpha\gamma \to \beta\gamma$.
Transitivity
If $\alpha \to \beta$ and $\beta \to \gamma$, then $\alpha \to \gamma$.
Union (derived)
If $\alpha \to \beta$ and $\alpha \to \gamma$, then $\alpha \to \beta\gamma$.
Decomposition (derived)
If $\alpha \to \beta\gamma$, then $\alpha \to \beta$ and $\alpha \to \gamma$.
Pseudo-transitivity (derived)
If $\alpha \to \beta$ and $\gamma\beta \to \delta$, then $\alpha\gamma \to \delta$.

Canonical proof layout; prove pseudo-transitivity

Given: $\alpha \to \beta$ and $\gamma\beta \to \delta$. To prove: $\alpha\gamma \to \delta$.

Numbered proof
1. α → β (given)
2. αγ → βγ (augmentation with γ, from 1)
3. γβ → δ (given)
4. βγ → δ (commutativity of set, from 3)
5. αγ → δ (transitivity, from 2 and 4) ∎

Notice: each step numbered on the left, FD in the middle, justification in parentheses on the right naming the rule and which prior steps were used. Close with $\blacksquare$ (or "QED" or "$\square$").

Ex D: prove pseudo-transitivity // number; cite rule; cite step

Given: α → β , γβ → δ
Show: αγ → δ
 1.  α → β                 (given)
 2.  αγ → βγ               (augmentation, from 1)
 3.  γβ → δ                (given)
 4.  βγ → δ                (commutativity of set, from 3)
 5.  αγ → δ                (transitivity, from 2 and 4)      ∎
→ each step earns partial marks; never skip the rule name.
Grading tip. Even if the final FD you derive is wrong, partial marks are awarded for each correctly cited rule. Always name the rule and list the antecedent step numbers. "By augmentation, from step 2" is worth half a mark even on a failed proof.

Part E: BCNF Decomposition Tree on Paper

Draw it like a family tree. The original relation $R$ is the root at the top. Each BCNF violation splits it into two children; label the arrow with the violating FD. If a child is still not in BCNF, it splits again. Boxed leaves are in BCNF; branch everything else.

Decomposition tree for A4 Q6

                  R = (A, B, C, D, E)
                  F = {A→BC, CD→E, B→D, E→A}
                  CKs: {A}, {BC}, {CD}, {E}
                  Violation: B → D (B not super key)
                         |
                         | split via B → D
                         v
              +-----------------+
              |                 |
              v                 v
       [R1 = (B, D)]     R2 = (A, B, C, E)
       FDs: {B → D}      FDs: {A → BC, E → A, BC → E}
       CK: {B}           CKs: {A}, {BC}, {E}
       ✓ BCNF            ✓ BCNF

Both leaves are boxed (marked with ✓ BCNF); no further decomposition needed. The arrow is annotated with the violating FD that caused the split.

Ex E: BCNF decomposition tree (A4 Q6 shape) // label arrows with the FD that split

                  R = (A, B, C, D, E)
                  F = { A→BC , CD→E , B→D , E→A }
                  CKs: {A}, {BC}, {CD}, {E}

                  Violation: B → D     (B not superkey)
                            |
                            |  split on B → D
                            v
                  +---------+---------+
                  |                   |
                  v                   v
              ┌────────┐        R2 = (A, B, C, E)
              │ R1 =   │        FDs: { A→BC , E→A , BC→E }
              │ (B, D) │        CKs: {A}, {BC}, {E}
              │ B → D  │        ✓ BCNF
              │ ✓ BCNF │
              └────────┘
→ box every BCNF leaf; label every split arrow with its FD.

Why the tree diagram earns marks

The tree makes the decomposition visible; the grader can trace every step without squinting. It also forces you to label the violating FD at each split, which is explicitly required for full marks. Students who just write "R → R1 and R2" without the FD label lose 1-2 marks.

Part F: Time Budget for 120 Marks in 180 Minutes

Base rate: ~1.5 minutes per mark.

Some blocks are faster (concept questions: 3 min per 2-mark question) and some slower (BCNF showpiece: ~1.7 min/mark).

The BCNF showpiece is the time sink

That 25-minute block looks long, but it contains five sub-tasks: find candidate keys, identify violation, decompose, check lossless, check dependency preservation. Each takes 4-6 minutes. If you start this at the 150-minute mark with only 10 minutes left, you cannot finish. Schedule it so it ends by the 175-minute mark. Leave at least 5 minutes true buffer for review.

Part G: Exam Day Tactical Tips

Before you start writing

  1. Read every question twice. Underline the key verbs: find, list, decompose, prove, draw, map.
  2. Note the mark value next to each question. 3-mark questions get 4-5 minutes max; anything longer means you are overthinking.
  3. Flip through the whole exam first (30 seconds). Spot any surprise topics and mentally allocate time.

While writing

  1. Attempt every question. Partial credit is real; even one correct line earns marks. A blank earns zero.
  2. Normalization questions: at the top of your answer space, write out candidate keys and FDs before doing anything else; you will refer to them constantly.
  3. ER questions: pencil first, ink last. Never start in pen. Leave space between entities; crowded diagrams lose marks.
  4. Show work on FD proofs and closures; wrong answers with correct reasoning still earn marks.
  5. Don't over-join in SQL. If a question does not need the company table, do not include it. Over-joining is a common reason to lose 0.5-1 marks.
  6. Use basic RA operators ($\sigma, \pi, \bowtie, \cup, -, \div$) unless the question explicitly allows extended operators ($\gamma$ for aggregation, outer joins). The prof's grading key uses the basic set.

At the end

  1. Never leave 10 minutes on the clock unused. Use them to revisit the hardest question.
  2. Re-read every SQL query you wrote. Check: DISTINCT where needed? HAVING not WHERE for aggregates? Both join predicates present?
  3. Re-read your BCNF decomposition. Lossless check cited with the specific FD. Dependency check done FD-by-FD.
  4. Verify ER diagrams. Every relationship has a cardinality. Every entity has a PK underlined. Weak entities have double borders.

The 10-commandment cheat-card (memorize)

  1. Lowercase SQL keywords; one clause per line.
  2. Two negations (NOT EXISTS ... NOT EXISTS) for "every" / universal quantification.
  3. HAVING filters groups; WHERE filters rows.
  4. Division $\div$ when the question says "all" or "every" across two sets.
  5. Rename $\rho$ before any self-join.
  6. Candidate keys via attribute closure; no guessing.
  7. BCNF violation = non-super-key determinant; decompose on the violation.
  8. Lossless check: shared attributes determine a fragment.
  9. Dependency-preservation check: every FD lives entirely in some fragment.
  10. ER: pencil first, ink last; label every cardinality.

Glossary

Armstrong's axioms
Reflexivity, augmentation, transitivity; the three sound-and-complete rules for inferring FDs.
Decomposition tree
Family-tree sketch of BCNF splits; leaves boxed when in BCNF.
Over-join
Including a table that is not needed for the query; costs marks for imprecision.
Partial credit
Marks awarded for correct sub-steps even when the final answer is wrong.
Total participation
Drawn with a double line; every entity instance must participate.
Discriminator
Local attribute that distinguishes instances of a weak entity within its owner group.
Showpiece question
The 15-mark BCNF decomposition; the biggest single-question mark cluster on the final.
