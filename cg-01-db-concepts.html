Database Concepts

Schemas, DBMS internals, keys, and integrity; the vocabulary every later page leans on.

Part A: Schema vs Instance

Physical analogy; a filing cabinet.

Picture a metal filing cabinet in a law office. The labels on each drawer (CLIENTS, CASES, INVOICES), the dividers inside, and the blank pre-printed forms that everyone fills out; that entire structure is the schema. It rarely changes; maybe once a year someone adds a new drawer or a new field to the intake form. The actual papers stuffed into the folders right now, the specific clients and case numbers and dollar amounts, that is the instance. It changes constantly: papers get added, shredded, refiled, updated. Two months from now the cabinet will hold a completely different instance, but the schema will be almost identical.

So when we talk about a database "design," we mean the schema. When we talk about the "data," we mean the current instance. Relational algebra queries and SQL queries are all written against the schema; they return whatever instance happens to be there when you run them.
Definition: Schema and Instance.
Schema
The logical structure of the database. Tables, attributes (columns), domains (data types), and constraints. Written once, changes rarely. Also called the intension.
Instance
The actual set of tuples (rows) sitting in the tables at a particular moment. Changes with every INSERT, UPDATE, DELETE. Also called the extension.
Three levels
Physical (bytes on disk, indexes, blocks) → Logical (tables and columns) → View (what a particular user sees). Each level is a schema of its own.

Schema vs Instance for the library database

Step 1: Write the schema

Three tables, written in the standard "relation-name(attr1, attr2, ...)" form with the primary key underlined conceptually:

employee(empid, name, office, age)
books(isbn, title, authors, publisher)
borrow(empid, isbn, borrow_date)
Step 2: Show one possible instance
employee (instance on April 17, 2026)
empidnameofficeage
E01AdaD-21034
E02BorisD-21429

Tomorrow Ada might leave and a new row might appear; the schema stays identical, only the instance changes.

Why This Matters

Every exam question that says "consider the relation $R(A,B,C)$ with the following tuples" is asking you to reason about both. Constraints live in the schema; they must be checked against every instance. When you write a relational algebra expression or an SQL query, you are describing a computation on the schema; the DBMS runs it on whatever instance is current.

Part B: Inside the DBMS

Physical analogy; a restaurant kitchen.

A customer (the user) hands a server a written order (an SQL query). The server does not cook; she passes the slip to the expeditor who reads it, checks whether the kitchen can actually make that dish, plans which stations need to do what, and in which order. That is the query processor. The cooks who actually grab ingredients from the walk-in fridge are the storage manager; they deal with raw materials, not recipes. Pinned to the wall is a menu listing every dish the kitchen knows how to make, every ingredient in stock, and every allergy note; that is the data dictionary. The whole system works because each specialist only talks to the layers above and below.
Definition: The three main DBMS subsystems.
Storage Manager
Handles the physical side: file I/O, buffer pool (memory pages), indexes, transactions, locks, logging for recovery. Knows about disk blocks, never about SELECT statements.
Query Processor
Parses SQL, validates it against the dictionary, builds a query plan, optimizes the plan (picks join order, index usage), and executes it by calling into the storage manager. This is the compiler + runtime for queries.
Data Dictionary
Also called the system catalog. A set of tables that describe every other table: relation names, attribute names and domains, primary keys, foreign keys, constraints, views, user privileges. The DBMS uses it to validate queries before running them.
Terminology. Different textbooks split these further; the DDL compiler, DML compiler, authorization manager, and transaction manager are all sub-components. For CPSC 3660 the storage manager / query processor / data dictionary split is the level of detail the prof tests.

Tracing a SELECT through the DBMS

A user submits: select name from employee where age > 30;

Step 1: Parser

The query processor's parser tokenizes the SQL and builds a parse tree. "Is this syntactically a valid SELECT?" Yes.

Step 2: Dictionary lookup

The parser asks the data dictionary: "Does a relation called employee exist? Does it have an attribute name? Does it have an attribute age? What are their domains?" The dictionary confirms: yes, age is an integer; comparing it to 30 is legal.

Step 3: Optimizer

The optimizer reads statistics from the dictionary (how many rows, whether age is indexed) and picks a plan. With no index, the plan is "sequentially scan employee, filter rows with age > 30, project name."

Step 4: Executor calls storage manager

The executor asks the storage manager for pages of the employee file. The storage manager fetches blocks from disk (or from the buffer pool if cached), hands them back. The executor applies the filter, projects, streams results to the user.

Result: two specialists, one catalog, one query answered.

Why This Matters

Knowing these three pieces lets you reason about performance and errors. "Table not found" is a dictionary error. "Disk I/O is slow" is a storage manager problem. "My query is slow even though the table is small" is usually a query processor problem (bad plan). The exam sometimes asks you to name the component responsible for a given task; keep the split clear.

Part C: The Data Dictionary, Up Close

Physical analogy; the card catalog in an old library.

Walk into a pre-1990 library and you find a wooden cabinet full of index cards. Each card describes one book: title, author, call number, where it sits on the shelves, what subjects it covers. The cabinet is not the books; it is a description of the books. The librarian never walks the stacks blindly; she opens the card catalog first, locates the call number, then goes. Without the catalog, finding anything would require scanning every shelf. The data dictionary is the same idea, but for tables and columns; the DBMS checks the catalog before every query, and keeps it up to date whenever the schema changes.
What the dictionary stores (typical contents).
Relations
Name of each table, its owner, its storage location on disk.
Attributes
For every column: name, domain (data type), length, NULL allowed?, default value.
Keys
Which attribute(s) form the primary key; which candidate keys exist.
Foreign keys
Which attribute in relation $R$ references which key in relation $S$; what to do on delete/update (cascade, restrict, set null).
Constraints
CHECK conditions, NOT NULL flags, UNIQUE declarations.
Views
Definitions of virtual tables (the SQL that defines them).
Statistics
Row counts, value distributions, histogram buckets, index selectivity; used by the optimizer.
Privileges
Which users can SELECT/INSERT/UPDATE/DELETE on which relations.
Self-describing property. The dictionary is itself stored in relations. In PostgreSQL you can literally write select * from pg_tables; and get back rows describing every table, including the pg_tables table itself. This is called a meta-database.

Exam Pattern

Expect a short-answer question like "What is the data dictionary and why does the DBMS need it?" A good answer names three uses: (1) validating queries (does the table exist? is this column an integer?), (2) enforcing constraints (is this FK target still there?), (3) feeding the optimizer statistics. Do not just say "it stores metadata"; show that you know what the DBMS does with it.

Part D: Keys; Super, Candidate, Primary

Physical analogy; pieces of ID in your wallet.

You have several things that can uniquely identify you: your driver's licence, your passport, your SIN, your student card plus date-of-birth combo. Any of these, alone or bundled, can pick you out of a crowd of a billion; each one is a candidate key. If I hand you your driver's licence AND your coffee cup, the combination still identifies you uniquely, but the coffee cup is superfluous; that bundle is a super key (identifies uniquely but contains extras). From among the candidate keys, the government picks ONE as the official identifier; for Canadians it is often the SIN. That chosen one is the primary key. The rejected candidates (passport, driver's licence) become alternate keys; still unique, still enforced, but not the designated one.
Definition: the three flavors of key.
Super key
Any set of attributes $K \subseteq R$ such that no two tuples of $R$ can agree on all of $K$ at the same time. May contain unnecessary attributes.
Candidate key
A super key that is minimal: remove any attribute and it stops being a super key. No redundant columns.
Primary key
The one candidate key chosen by the designer as the official row identifier. Must be NOT NULL and must be unique. Every relation has exactly one.
Foreign key
An attribute (or set) in relation $R$ whose values must appear as the primary key of some row in relation $S$. Enforces cross-table consistency.

Finding every super key and candidate key of employee(empid, name, office, age)

Step 1: Identify the uniqueness guarantee

The university assigns each employee a unique empid. So {empid} uniquely identifies a tuple. That is our starting point.

Step 2: Build super keys by adding anything

Any superset of a super key is still a super key. So:

{empid}, {empid, name}, {empid, office}, {empid, age},
{empid, name, office}, {empid, name, age}, ..., {empid, name, office, age}

All of these are super keys because they all contain empid.

Step 3: Prune down to candidate keys

A candidate key must be minimal. {empid} alone already works, so any super key containing it AND something else fails the minimality test. Only {empid} survives as a candidate key.

Candidate keys: {empid}. Primary key: empid.

Step 4: What if names were also unique?

If the problem stated "no two employees share a name," then {name} would also be a candidate key. We would then have TWO candidate keys: {empid} and {name}. The designer picks one (usually empid) as the primary key; the other becomes an alternate key.

This is where exam questions like "list all candidate keys of R" come from; see Midterm Q2 and A1 Q3.

Exam Pattern (Midterm Q2, A1 Q3)

The prof loves asking you to define each key type and give an example, then to enumerate candidate keys for a relation with stated uniqueness constraints. Always: (1) start from any guaranteed-unique attribute, (2) combine attributes when no single one is unique, (3) prune non-minimal sets. If the question says "state the primary key," pick the most stable, shortest candidate key and justify the choice in one sentence.

Part E: Integrity Constraints

Physical analogy; rules printed on a paper form.

When you fill out a passport application the form itself enforces rules: the date-of-birth box refuses letters, the "emergency contact phone" field cannot be left blank, the "citizenship" field must match a code from an approved list. These rules live on the form, not in the person filling it out; so every applicant is forced to obey them. Integrity constraints are exactly that; they are glued to the schema and the DBMS refuses any insert or update that would break them. The database never lets invalid data in, no matter which user or application tries.
The four main integrity constraint types.
Domain
Each attribute value must come from its declared domain. age INTEGER cannot hold "twelve"; borrow_date DATE cannot hold "banana".
Entity integrity
The primary key of any relation must never be NULL. A row with no identifier cannot exist.
Referential integrity
Every foreign-key value must either be NULL or match an existing primary-key value in the referenced relation. No orphan references.
User-defined (CHECK)
Arbitrary conditions: CHECK (age >= 0), CHECK (balance >= 0), CHECK (status IN ('active','inactive')).

Referential integrity in the library schema

Step 1: Identify the FK

In borrow(empid, isbn, borrow_date), the attribute empid is a foreign key referencing employee(empid), and isbn is a foreign key referencing books(isbn).

Step 2: Attempt a bad insert
insert into borrow values ('E99', '9780-XYZ', '2026-04-17');

If E99 does not exist in employee, referential integrity is violated. The DBMS rejects the insert; the row never lands in borrow.

Step 3: Delete a referenced row
delete from employee where empid = 'E01';

If any row in borrow has empid = 'E01', deleting the employee would leave orphans. The DBMS must do one of three things, depending on the FK's ON DELETE action:

  • RESTRICT (or NO ACTION): refuse the delete; throw an error.
  • CASCADE: also delete the matching rows in borrow.
  • SET NULL: keep the borrow rows, but set their empid to NULL (only legal if the FK column permits NULLs).

CASCADE is the most aggressive option; it propagates the delete down the dependency chain.

Part F: Cascading Deletes

Physical analogy; tipping over the first domino.

You have a chain of dominoes arranged so each one rests partially on the next. Knock down the first and the rest fall automatically in order. ON DELETE CASCADE is exactly that: when the parent row falls, every dependent row in a child table falls too, and if those rows are parents to a third table with another CASCADE, that falls next, and so on. It is powerful and dangerous; a single DELETE against employee can wipe rows from three or four tables in one statement.
Rule: ON DELETE CASCADE behavior.

If child.fk REFERENCES parent.pk ON DELETE CASCADE, then deleting a row from parent automatically deletes every row in child whose fk matched. The cascade is transitive; if grandchild.fk2 REFERENCES child.pk ON DELETE CASCADE, those rows go too.

Tracing a cascade

Schema with cascading FKs:

create table employee (
  empid char(5) primary key, name varchar(40)
);

create table borrow (
  empid char(5), isbn char(13), borrow_date date,
  primary key (empid, isbn),
  foreign key (empid) references employee(empid)
    on delete cascade
);
Initial state

employee: (E01, Ada), (E02, Boris).
borrow: (E01, 978-A, 2026-03-01), (E01, 978-B, 2026-03-02), (E02, 978-A, 2026-04-10).

Issue the command
delete from employee where empid = 'E01';
What happens

(1) Ada's row is removed from employee. (2) Both borrow rows with empid='E01' are automatically removed. Boris and his one borrow row are untouched.

Final: employee=[(E02,Boris)], borrow=[(E02, 978-A, 2026-04-10)].

Why This Matters

Cascades are a design decision, not a syntax detail. Choose CASCADE when child rows have no meaning without the parent (an employee's borrow history is meaningless if the employee is gone). Choose RESTRICT when the child rows are important history that should prevent casual deletes (do not cascade-delete a customer who has invoices; block the delete and force an explicit cleanup). Choose SET NULL rarely; usually only when the reference is optional.

Exam Pattern

The prof has asked "what happens if we delete employee E01 from the schema, given the FK constraint shown?" Your answer should name the CASCADE / RESTRICT / SET NULL rule, then walk through the specific rows that disappear. Do not just say "the child rows are deleted"; name them.

Glossary

Schema
Logical structure of the database (tables, columns, domains, constraints). Changes rarely.
Instance
Current set of tuples sitting in the tables. Changes with every DML statement.
DBMS
Database Management System; the software (PostgreSQL, Oracle, etc.) that implements the relational model.
Storage manager
DBMS subsystem that handles disk I/O, buffer pool, locking, logging, transactions.
Query processor
DBMS subsystem that parses, optimizes, and executes SQL.
Data dictionary / system catalog
Tables that describe the schema; used by the DBMS for validation and optimization.
Super key
Any attribute set that uniquely identifies tuples. May contain extras.
Candidate key
A minimal super key; no attribute can be removed.
Primary key
The chosen candidate key. NOT NULL, UNIQUE, exactly one per relation.
Foreign key
Attribute(s) in one relation whose values must match a primary key in another relation.
Entity integrity
Primary keys must never be NULL.
Referential integrity
Foreign keys must either be NULL or point to an existing primary key.
ON DELETE CASCADE
When the referenced row is deleted, all referencing rows are deleted too. Transitive.
ON DELETE RESTRICT
Refuse the delete if any referencing rows exist.
ON DELETE SET NULL
Set the FK columns in referencing rows to NULL.
