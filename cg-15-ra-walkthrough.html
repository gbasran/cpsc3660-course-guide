8 Relational Algebra Problems, Side-by-Side with SQL

Each problem is solved in RA on the left, SQL on the right. Division, grouping, and set ops all walked through.

Physical analogy; looking at a blueprint and a photograph of the same house.

An architect reads the blueprint and understands the geometry; a buyer looks at a photograph and sees the result. For every RA expression there is a corresponding SQL query that produces the same rows; the RA is the blueprint (compact, algebraic) and the SQL is the photograph (procedural, verbose). Training your brain to flip between them is the whole point of this page; you will face an exam question asking for one and sometimes both.

Problem 1: Basic Selection + Projection + Join (Library)

Problem (2 marks). Find the names of employees who have borrowed a book published by "Springer".

RA expression:

$$\pi_{name}\bigl(\sigma_{publisher='Springer'}\,(employee \bowtie borrow \bowtie books)\bigr)$$

Equivalent SQL
select distinct e.name
from employee e, borrow b, books k
where e.empid = b.empid
  and b.isbn = k.isbn
  and k.publisher = 'Springer';

Walkthrough

Step 1: Join the three relations on shared attributes

Natural join chains: employee $\bowtie$ borrow on empid; that result $\bowtie$ books on isbn.

Step 2: Select where publisher = Springer

Apply $\sigma$ with the publisher predicate. This filters the combined tuples.

Step 3: Project name

$\pi_{name}$ keeps only the name column; duplicates are removed automatically in RA (set semantics).

Mirrors A1 Q7(a); canonical warm-up

Problem 2: Natural Join Exposure (Library)

Problem (2 marks). List every (employee name, book title) pair where the employee has borrowed that book.

RA expression:

$$\pi_{name,\,title}\,(employee \bowtie borrow \bowtie books)$$

Equivalent SQL
select e.name, k.title
from employee e, borrow b, books k
where e.empid = b.empid
  and b.isbn = k.isbn;

Walkthrough

Step 1: Chain natural joins

Employee and borrow share empid; borrow and books share isbn. The natural join handles both automatically.

Step 2: Project the two asked-for columns

$\pi_{name, title}$; the projection list names the attributes you keep.

Note on ambiguity

If authors in books overlapped with another attribute, natural join would inadvertently match on it. Always check which attributes are shared before writing $\bowtie$.

Mirrors A1 Q7(a); A2 Q1(a)

Problem 3: Division (Library) — THE HARD ONE

Problem (3-4 marks; hard; guaranteed). Find the names of employees who have borrowed every book published by "Springer".

RA expression:

$$R_1 = \pi_{empid,\,isbn}\,(borrow)$$

$$R_2 = \pi_{isbn}\bigl(\sigma_{publisher='Springer'}(books)\bigr)$$

$$R_3 = R_1 \div R_2 \quad (\text{employees who borrowed all Springer books, by } empid)$$

$$\text{Final} = \pi_{name}\bigl(employee \bowtie R_3\bigr)$$

Equivalent SQL (NOT EXISTS form)
select e.name
from employee e
where not exists (
  select 1 from books k
  where k.publisher = 'Springer'
  and not exists (
    select 1 from borrow b
    where b.empid = e.empid and b.isbn = k.isbn
  )
);

Walkthrough of division itself

Definition of $\div$

If $R_1$ has schema $(X, Y)$ and $R_2$ has schema $(Y)$, then $R_1 \div R_2$ returns the set of $X$-values in $R_1$ that are paired with every $Y$-value in $R_2$. In other words, for each candidate $x$, we check that for every $y \in R_2$, the tuple $(x,y)$ appears in $R_1$; if so, $x$ survives.

Step 1: Build the "who borrowed what" table

$R_1 = \pi_{empid, isbn}(borrow)$; the set of all (employee, book) pairs that actually happened.

Step 2: Build the "which books must be borrowed" set

$R_2 = \pi_{isbn}(\sigma_{publisher='Springer'}(books))$; all Springer ISBNs.

Step 3: Divide

$R_3 = R_1 \div R_2$; empid values in $R_1$ paired with every ISBN in $R_2$.

Step 4: Join to get names

$R_3$ has only empid values; join back with employee and project name.

Express $\div$ with basic operators (if prof demands)

$R_1 \div R_2 = \pi_X(R_1) - \pi_X\bigl((\pi_X(R_1) \times R_2) - R_1\bigr)$; "all candidate X-values, minus those with an unfulfilled Y-pair."

Mirrors A1 Q7(b); A2 Q1(b); THE division question

Problem 4: Rename + Self-Join (Bank)

Problem (3 marks; medium). Find the account numbers with the second-smallest balance in the account relation.

RA expression:

$$R = \rho_{A_1}(account),\quad S = \rho_{A_2}(account)$$

$$\text{smallest} = \pi_{A_1.balance}\bigl(R\bigr) - \pi_{A_1.balance}\bigl(\sigma_{A_1.balance \gt A_2.balance}(R \times S)\bigr)$$

$$\text{second} = \pi_{A_2.account\_number}\bigl(\sigma_{A_2.balance = (\text{smallest after removing itself})}(\ldots)\bigr)$$

More cleanly, the "second smallest balance value" is:

$$b_2 = \pi_{balance}(account) - \{\min\ balance\} \text{ then min again}$$

Equivalent SQL
select account_number, balance
from account
where balance = (
  select min(balance) from account
  where balance > (select min(balance) from account)
);

Walkthrough

Step 1: Why rename is needed

Whenever you join a relation to itself in RA, you must rename one copy with $\rho$ so that the two sets of attributes are distinguishable. Without rename, $account \times account$ has duplicate attribute names and cannot be filtered.

Step 2: Find the smallest via "remove those with anything smaller"

The smallest balance is the balance for which no other row has a smaller balance. Self-join and filter where $A_1.balance \lt A_2.balance$; subtract.

Step 3: Apply the trick again to the set minus smallest

Second smallest = smallest of the set obtained by removing the smallest.

Mirrors A1 Q5(b); classic rename exercise

Problem 5: Grouping + Aggregation in RA (Library)

Problem (3 marks; hard). Find the names of authors whose books have been borrowed by at least 3 distinct employees.

RA expression (using $\gamma$ for aggregation):

$$T_1 = employee \bowtie borrow \bowtie books$$

$$T_2 = {}_{authors}\gamma_{\text{count-distinct}(empid)\to cnt}\,(T_1)$$

$$\text{Final} = \pi_{authors}\bigl(\sigma_{cnt \geq 3}(T_2)\bigr)$$

Equivalent SQL
select k.authors
from employee e, borrow b, books k
where e.empid = b.empid and b.isbn = k.isbn
group by k.authors
having count(distinct e.empid) >= 3;

Walkthrough

Step 1: Join everything that is needed

Three-way join to connect authors (in books) to employees (in employee) via borrow.

Step 2: Group on the "by author" axis, aggregate count-distinct

$_{authors}\gamma_{\text{count-distinct}(empid)}$; the subscript before $\gamma$ is the grouping attribute list.

Step 3: Filter groups with $\sigma$

$\sigma_{cnt \geq 3}$ on the grouped result; equivalent to SQL's HAVING clause.

Mirrors A1 Q7(c); Midterm Q8(c)

Problem 6: Multi-Level Grouping (Library)

Problem (4 marks; hard). For each publisher, find the employees who have borrowed more than 5 of that publisher's books.

RA expression:

$$T = employee \bowtie borrow \bowtie books$$

$$G = {}_{publisher,\,empid,\,name}\gamma_{\text{count}(isbn)\to n}\,(T)$$

$$\text{Final} = \pi_{publisher,\,name}\bigl(\sigma_{n \gt 5}(G)\bigr)$$

Equivalent SQL
select k.publisher, e.name
from employee e, borrow b, books k
where e.empid = b.empid and b.isbn = k.isbn
group by k.publisher, e.empid, e.name
having count(b.isbn) > 5;

Walkthrough

Step 1: Group on multiple attributes

Group by (publisher, empid, name). Each distinct combination becomes a group.

Step 2: Aggregate and filter

Count ISBNs per group; filter groups with $n \gt 5$.

Why empid is in GROUP BY even when we only output name

If two employees share a name, grouping only by name would merge their counts. Grouping by empid keeps them separate; projecting name at the end is safe because empid determines name.

Mirrors A1 Q7(d)

Problem 7: Set Operations (Bank)

Problem (2 marks). Find the names of customers who have an account but no loan.

RA expression:

$$\pi_{customer\_name}(depositor) - \pi_{customer\_name}(borrower)$$

Equivalent SQL
(select customer_name from depositor)
except
(select customer_name from borrower);

Walkthrough

Step 1: Project before the difference

Both operands of $-$ must be union-compatible (same attributes). Projecting customer_name from each makes them so.

Step 2: Set difference is the RA match for "but not"

Set difference $-$ directly expresses exclusion. No correlated subquery needed in RA.

Mirrors A3 Q2

Problem 8: RA Insert via Union (Midterm Q9(c) style)

Problem (3 marks; medium). An employee (E05, Dora, D-220, 31) borrows book 978-NEW on 2026-04-17. Write RA assignments that simultaneously add the employee to employee, add the book (if not already there), and add the borrow record.

RA assignments:

$$employee \leftarrow employee \cup \{(E05,\ Dora,\ D\text{-}220,\ 31)\}$$

$$books \leftarrow books \cup \{(978\text{-}NEW,\ title,\ authors,\ publisher)\}$$

$$borrow \leftarrow borrow \cup \{(E05,\ 978\text{-}NEW,\ 2026\text{-}04\text{-}17)\}$$

Equivalent SQL
insert into employee values ('E05', 'Dora', 'D-220', 31);
insert into books values ('978-NEW', 'New Title', 'Author Name', 'Publisher');
insert into borrow values ('E05', '978-NEW', '2026-04-17');

Walkthrough

Step 1: RA INSERT is just union with a singleton

$r \leftarrow r \cup \{t\}$ is the algebraic form of INSERT; the arrow denotes assignment (reassign $r$ to include the new tuple).

Step 2: Order of operations matters for FKs

Insert the parent rows before the child; otherwise the FK constraint on borrow.empid and borrow.isbn would fail.

Step 3: Check for duplicates

RA union is set-based; if the tuple is already in the relation, the union does nothing. In SQL, duplicate PKs throw an error instead.

Mirrors Midterm Q9(c)

The RA-to-SQL flip drill

Pick a problem at random. Cover the SQL side; write it yourself from the RA. Then cover the RA side; derive the RA from the SQL. If you can do both directions for problems 3, 5, and 6, you can handle any RA question the prof writes. See also Page 07 for the systematic translation rules.

Glossary

$\sigma$ (sigma)
Selection; keeps tuples satisfying a predicate.
$\pi$ (pi)
Projection; keeps the listed attributes; duplicates removed.
$\bowtie$ (bowtie)
Natural join; combines tuples matching on shared attributes.
$\rho$ (rho)
Rename; relabels a relation or its attributes.
$\gamma$ (gamma)
Generalized projection with aggregation; equivalent to SQL GROUP BY + aggregate.
$\div$ (division)
Returns the X-values paired with every Y-value in the divisor.
$\cup, \cap, -$
Union, intersection, set difference; operands must be union-compatible.
$\times$ (Cartesian product)
Pairs every tuple of $R$ with every tuple of $S$.
$\leftarrow$ (assignment)
RA form of INSERT / UPDATE; reassigns a relation variable.
