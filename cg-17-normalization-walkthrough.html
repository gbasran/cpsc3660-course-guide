3 Full Normalization Walkthroughs

The A4 Q6 gold standard, the lossless bank example, and the lossy-join counter-example. Every algebra step shown.

Physical analogy; splitting a cluttered drawer into two neat drawers.

You have one drawer full of socks, underwear, and T-shirts all jumbled together. Every morning you waste time digging. You split it into two drawers: one for socks, one for shirts + underwear. That split works if every item still has a home (nothing disappears; that is lossless), and if the "find me a black sock that matches this one" rule can still be enforced without looking across drawers (that is dependency preservation). Normalization is this drawer-splitting exercise, done on tables, driven by functional dependencies.
The BCNF pipeline, in one breath.
  1. Compute candidate keys of $R$ under $F$.
  2. For each non-trivial FD $\alpha \to \beta$ in $F$, check whether $\alpha$ is a super key.
  3. If not, $\alpha \to \beta$ is a BCNF violation; decompose $R$ into $R_1 = (\alpha \cup \beta)$ and $R_2 = R - \beta$.
  4. Recurse on $R_1$ and $R_2$ until every fragment is in BCNF.
  5. Check lossless join ($R_1 \cap R_2 \to R_1$ or $\to R_2$).
  6. Check dependency preservation (each FD lives in some fragment).

Walkthrough 1: A4 Q6 (BCNF Gold Standard)

Given. $R = (A, B, C, D, E)$ with $F = \{A \to BC,\ CD \to E,\ B \to D,\ E \to A\}$.
Tasks. (a) Find all candidate keys. (b) Check whether $R$ is in BCNF. (c) If not, decompose into BCNF. (d) Is the decomposition lossless? (e) Is it dependency-preserving?

Part (a): candidate keys via attribute closure

Check $\{A\}^+$

Start: $\{A\}$. Apply $A \to BC$: $\{A,B,C\}$. Apply $B \to D$: $\{A,B,C,D\}$. Apply $CD \to E$: $\{A,B,C,D,E\}$.

$\{A\}^+ = R$; so $\{A\}$ is a candidate key.

Check $\{BC\}^+$

Start: $\{B,C\}$. Apply $B \to D$: $\{B,C,D\}$. Apply $CD \to E$: $\{B,C,D,E\}$. Apply $E \to A$: $\{A,B,C,D,E\}$.

$\{BC\}^+ = R$; $\{BC\}$ is a candidate key. Neither $B$ alone nor $C$ alone yields $R$, so it is minimal.

Check $\{CD\}^+$

Start: $\{C,D\}$. Apply $CD \to E$: $\{C,D,E\}$. Apply $E \to A$: $\{A,C,D,E\}$. Apply $A \to BC$: $\{A,B,C,D,E\}$.

$\{CD\}^+ = R$; $\{CD\}$ is a candidate key.

Check $\{E\}^+$

Start: $\{E\}$. Apply $E \to A$: $\{A,E\}$. Apply $A \to BC$: $\{A,B,C,E\}$. Apply $B \to D$: $\{A,B,C,D,E\}$.

$\{E\}^+ = R$; $\{E\}$ is a candidate key.

Collect the candidate keys

Candidate keys: $\{A\},\ \{BC\},\ \{CD\},\ \{E\}$. Prime attributes: $A, B, C, D, E$ (all of them). Non-prime: none.

Part (b): check BCNF

Check each FD in $F$

Part (c): decompose using the violation

Apply the BCNF decomposition rule

Given violation $\alpha \to \beta$ where $\alpha = B$, $\beta = D$. Form:

$R_1 = \alpha \cup \beta = \{B, D\}$

$R_2 = R - \beta = \{A, B, C, E\}$

Check $R_1 = (B, D)$ for BCNF

FDs projected onto $R_1$: $\{B \to D\}$. $B^+_{R_1} = \{B,D\} = R_1$, so $B$ is super key of $R_1$. BCNF satisfied. $\checkmark$

Check $R_2 = (A, B, C, E)$ for BCNF

Project FDs onto $R_2$: A → BC (both B and C are in $R_2$), E → A (A in $R_2$). Also derivable: $BC \to E$ in $R_2$ (since $BC \to D$ via $B \to D$, and $CD \to E$ chains; though $D$ is outside $R_2$, the implication $BC \to E$ still holds for $R_2$-tuples that extend to full $R$-tuples).

Candidate keys of $R_2$: $\{A\}$ (from $A \to BC$, then $BC \to E$, reaching $R_2$); $\{E\}$ (from $E \to A$); $\{BC\}$ (closes to $R_2$). Each determining set in the projected FDs is a super key. BCNF satisfied. $\checkmark$

Final decomposition

$R_1 = (B, D)$ and $R_2 = (A, B, C, E)$, both in BCNF.

Part (d): lossless-join check

Compute the common attributes

$R_1 \cap R_2 = \{B\}$.

Check $R_1 \cap R_2 \to R_1$ (or $\to R_2$) under $F$

Does $B \to R_1$? That is, does $B \to \{B, D\}$? Yes, since $B \to D \in F$ and trivially $B \to B$. So the shared attribute $B$ functionally determines the smaller fragment $R_1$.

Conclusion

The decomposition is lossless. $R_1 \bowtie R_2 = R$ for every legal instance; no spurious tuples. $\checkmark$

Part (e): dependency-preservation check

Track each FD
FDAttrsFragment containing all attrsPreserved?
$A \to BC$$\{A,B,C\}$$R_2 = \{A,B,C,E\}$Yes
$CD \to E$$\{C,D,E\}$Split! $D \in R_1$, $C,E \in R_2$No
$B \to D$$\{B,D\}$$R_1 = \{B,D\}$Yes
$E \to A$$\{A,E\}$$R_2 = \{A,B,C,E\}$Yes
Is $CD \to E$ recoverable via the union of projected FDs?

No single fragment contains $C$, $D$, and $E$ together. The projection of $F$ onto $R_1$ gives $\{B \to D\}$; onto $R_2$ gives FDs on $\{A,B,C,E\}$. Neither can re-derive $CD \to E$ without inspecting tuples across both fragments.

Conclusion

The decomposition is NOT dependency-preserving. The FD $CD \to E$ cannot be enforced by checking each fragment in isolation.

This is the classic BCNF trade-off; we always get lossless, but dependency preservation is not guaranteed. The prof explicitly flags this on A4.

Walkthrough 3: Lossy-Join Counter-Example (Week 14)

Given. $Employee(EmployeeID,\ Project,\ Branch)$ with FDs $EmployeeID \to Branch$ and $Project \to Branch$.
Decomposition. $R_1 = (EmployeeID, Branch)$, $R_2 = (Project, Branch)$.
Task. Show the decomposition is lossy; the natural join produces spurious tuples.

Theoretical check

Common attributes

$R_1 \cap R_2 = \{Branch\}$.

Does $Branch \to R_1$ or $\to R_2$ hold?

$Branch \to EmployeeID$? No; one branch can host many employees. $Branch \to Project$? No; one branch can host many projects. Neither direction holds.

Conclusion

Decomposition is LOSSY; natural join will invent spurious tuples.

Why it happened

Branch is the only shared attribute, and Branch does not determine either EmployeeID or Project. Multiple employees and multiple projects share Branch SJ; natural join combines every combination, not just the ones that were true.

Moral: for a decomposition to be lossless, the shared attributes must functionally determine at least one of the fragments. Here Branch determines neither, so the join invents pairings that never existed.

Decision tree: lossless vs dependency-preserving

Fast checks.
Lossless?
Compute $R_1 \cap R_2$; ask if it is a super key of $R_1$ OR of $R_2$ under $F$. If yes → lossless. If no → lossy.
Dependency-preserving?
For each FD $\alpha \to \beta$ in $F$, check whether there exists a fragment containing $\alpha \cup \beta$. If every FD lives entirely inside some fragment, preserved. If any FD is split across fragments, not preserved.
BCNF guarantees
Always lossless (by construction of the decomposition rule). May sacrifice dependency preservation. 3NF always preserves dependencies but may not reach BCNF.

What the grader wants to see

For 15 marks on the BCNF showpiece, write out: (1) each candidate-key closure line-by-line, (2) the FD violation table like the one in Walkthrough 1, (3) the decomposition with both fragments named, (4) the lossless check as an algebraic statement ($R_1 \cap R_2 = \{\ldots\}$, then show the FD that proves it), (5) the dependency-preservation check as a table per FD. If you skip the tables and just write "lossless, dep-preserving," you will lose half the marks.

Glossary

Attribute closure $\alpha^+$
The set of attributes functionally determined by $\alpha$ under $F$; computed by iteratively applying FDs until no change.
Candidate key
Minimal attribute set $K$ with $K^+ = R$.
Super key
Any attribute set whose closure is $R$; not necessarily minimal.
Prime attribute
Attribute appearing in at least one candidate key.
BCNF violation
Non-trivial FD $\alpha \to \beta$ with $\alpha$ not a super key.
Lossless decomposition
Natural-join back produces no spurious tuples; equivalent to common attrs determining one fragment.
Dependency preservation
Every FD in $F$ can be checked by inspecting one fragment; no cross-fragment joins needed.
Spurious tuple
A tuple that appears in $R_1 \bowtie R_2$ but was not in the original $R$.
