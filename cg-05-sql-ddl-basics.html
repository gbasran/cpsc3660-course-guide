SQL: DDL and Basics

CREATE TABLE, INSERT, and the SELECT/FROM/WHERE skeleton that handles 80 percent of queries.

Part A: CREATE TABLE; Declaring the Schema

Physical analogy; setting up a filing cabinet's drawer dividers and label rules.

Before anyone can file a single paper, someone has to go to the empty cabinet and set up the drawers: print the labels, insert the dividers, write down the rules ("every intake form must have a client ID; no two clients may share an ID; the phone field is optional but must contain digits only"). CREATE TABLE is exactly that setup step. You name the relation, list the columns with their domains, and attach every constraint you want enforced. After this runs, the table exists but is empty; you can INSERT rows into it.
Skeleton: CREATE TABLE.
create table <name> (
  <col1> <type> [not null] [default ...] [check (...)],
  <col2> <type> ...,
  primary key (<col> [, <col>]),
  foreign key (<col>) references <other_table>(<col>)
    [on delete cascade | on delete set null | on delete restrict],
  check (<condition>)
);
types
integer, numeric(p, s), char(n), varchar(n), date, boolean.
not null
Column cannot be NULL in any row.
primary key
Auto-implies NOT NULL and UNIQUE. Can span multiple columns (composite PK).
foreign key
Values must appear as the PK of the referenced table, or be NULL.
check
User-defined predicate; DBMS rejects rows that violate it.

Full library schema in SQL

Step 1: employee (standalone, no FKs)
create table employee (
  empid    char(5) primary key,
  name     varchar(40) not null,
  office   varchar(10),
  age      integer check (age >= 0 and age < 120)
);
Step 2: books (also standalone)
create table books (
  isbn     char(13) primary key,
  title    varchar(100) not null,
  authors  varchar(200),
  publisher varchar(50) not null
);
Step 3: borrow (two FKs, order matters)

You must create the two parent tables first; borrow references them.

create table borrow (
  empid       char(5),
  isbn        char(13),
  borrow_date date not null,
  primary key (empid, isbn),
  foreign key (empid) references employee(empid) on delete cascade,
  foreign key (isbn)  references books(isbn)    on delete restrict
);

Deleting an employee auto-cleans their borrow rows; deleting a book is blocked if anyone has borrowed it.

Exam Pattern (A2 Q6(a), Midterm Q8(d))

CREATE TABLE questions almost always include at least one FK with a cascade rule and one CHECK. Forgetting the on delete clause is partial credit at best. The prof also tests whether you order the CREATE statements correctly (parents before children). When in doubt, list all standalone tables first, then relationship tables.

Part B: INSERT

Physical analogy; dropping one form into a drawer.

INSERT writes one row at a time (or a batch) into the table. The DBMS checks every constraint on the way down: the PK must not already exist, every NOT NULL column must have a value, every FK must resolve, every CHECK must pass. If any check fails, the entire insert is aborted; no partial row is left behind.
Syntax: INSERT.
insert into <table> [(<col1>, <col2>, ...)]
  values (<v1>, <v2>, ...);

insert into <table> (<col-list>)
  select ... from ...;   -- bulk insert from a query

Populating the library

insert into employee values ('E01','Ada','D-210',34);
insert into employee values ('E02','Boris','D-214',29);

insert into books values ('9781','Intro to DB','Silberschatz','Penguin');
insert into books values ('9782','ER Modeling','Chen','MIT');

insert into borrow values ('E01','9781','2026-03-01');
insert into borrow values ('E02','9782','2026-03-02');

Notice all string literals use single quotes. Dates as 'YYYY-MM-DD' strings.

Part C: SELECT / FROM / WHERE; the Core Query

Physical analogy; a form letter asking the librarian for specific books.

SELECT ... FROM ... WHERE ... is a three-part question. FROM names which cabinet to look in. WHERE is the condition ("books published before 1950"). SELECT is what to hand back ("just the titles and ISBNs"). The DBMS reads WHERE first (filters rows), then FROM is already locked in, then SELECT picks columns. The order you write is SELECT-FROM-WHERE; the order the DBMS executes is FROM → WHERE → SELECT.
Skeleton: basic SELECT.
select <col-list> | *
from <table-list>
where <predicate>;
col-list
Comma-separated columns, or * for all.
table-list
One or more tables; if more than one, their rows are Cartesian-combined unless WHERE restricts.
predicate
Boolean condition using =, <>, <, <=, >, >=, and, or, not, between, in, like, is null.

Three basic queries

Query 1: all columns of young employees
select *
from employee
where age < 35;

Equivalent to $\sigma_{\text{age} \lt 35}(\text{employee})$.

Query 2: just names of employees in D-210
select name
from employee
where office = 'D-210';

Equivalent to $\pi_{\text{name}}(\sigma_{\text{office}=\text{'D-210'}}(\text{employee}))$.

Query 3: distinct offices
select distinct office
from employee;

Without DISTINCT you get duplicates; SQL's SELECT keeps bag semantics by default.

Part D: Multi-Table Queries (Joins)

Physical analogy; flipping back and forth between two open folders on your desk.

A single table rarely has everything. If you want "names of employees who borrowed a book," you start in borrow to see who borrowed what, then flip to employee to turn the empid into a name. SQL does this with a join. There are two syntactic flavors: the implicit join writes all tables in the FROM clause and puts the join condition in WHERE; the explicit join uses JOIN ... ON. Both produce the same result; the explicit form is clearer for humans.
Two ways to write a join.
-- implicit
select e.name, b.isbn
from employee e, borrow b
where e.empid = b.empid;

-- explicit (preferred)
select e.name, b.isbn
from employee e
join borrow b on e.empid = b.empid;
e, b
Table aliases; shorthand for the full names inside the query.
ON
Join condition; typically the FK = PK equality.

Three-table join: name, title, borrow_date

What we want

For every borrow event, show the employee's name, the book's title, and the borrow date.

Implicit form
select e.name, bk.title, br.borrow_date
from employee e, borrow br, books bk
where e.empid = br.empid
  and br.isbn = bk.isbn;
Explicit form
select e.name, bk.title, br.borrow_date
from employee e
join borrow br on e.empid = br.empid
join books bk on br.isbn = bk.isbn;

Both return the same tuples; the explicit form highlights each join predicate on its own line.

Exam Pattern (A2 Q7, Midterm Q7)

The prof writes both implicit and explicit forms in handouts; either is accepted on the exam as long as the join conditions are correct. For readability and partial credit, prefer the explicit JOIN ... ON form; the marker can see your join logic immediately.

Don't Over-Join

A common mistake: joining tables you do not need. If the question is "find the names of employees in D-210," do not join borrow and books; you only need employee. Every extra join adds tuples (and sometimes wrong duplicates) and hurts clarity. Rule of thumb: list the columns you need in the SELECT, then trace each one back to its source table, then include only those tables. If a table contributes neither a selection predicate nor an output column, it does not belong in FROM.

Part E: ORDER BY, DISTINCT, BETWEEN, LIKE, IN

Physical analogy; extra filters you add to a search.

These are the small operators you sprinkle into queries to shape the output. ORDER BY sorts; DISTINCT deduplicates; BETWEEN is shorthand for a range check; LIKE does pattern matching on strings; IN tests membership in a list. None of them change which tuples you conceptually want; they change how you write the predicate or how the result is presented.
Quick reference.
order by
order by col [asc|desc], col2 .... Default is asc. Appears after WHERE, before any LIMIT.
distinct
select distinct col, .... Removes duplicate rows from the result.
between
x between a and b is shorthand for x >= a and x <= b. Inclusive on both ends.
like
name like 'A%' matches names starting with A. % = any string, _ = any single char.
in
office in ('D-210','D-214','D-300') is shorthand for a disjunction of equalities.

Five small queries

Employees sorted by age descending
select name, age
from employee
order by age desc;
Employees aged 30 to 40 inclusive
select name
from employee
where age between 30 and 40;
Books whose title contains "data" (case-sensitive in most DBMSs)
select title
from books
where title like '%data%';

Anchored forms: 'data%' = starts with data; '%data' = ends with data.

Employees in one of three offices
select name
from employee
where office in ('D-210','D-214','D-300');
Distinct publishers in alphabetical order
select distinct publisher
from books
order by publisher;

These five constructs cover 90 percent of the syntax you need on the exam.

Why This Matters

The exam's early SQL questions are judged on precision of syntax. Miss a comma, swap AND for OR, use = where IS NULL was needed, and you lose marks. Practice writing these operators from memory: BETWEEN is inclusive, LIKE uses % and _, IN requires parentheses around the list, ORDER BY comes last. These tiny details cost points.

Glossary

DDL
Data Definition Language; CREATE TABLE, ALTER TABLE, DROP TABLE.
DML
Data Manipulation Language; INSERT, UPDATE, DELETE, SELECT.
NOT NULL
Column constraint rejecting NULL values.
PRIMARY KEY
Implies UNIQUE and NOT NULL; one per table.
FOREIGN KEY ... REFERENCES
Declares referential integrity and cascade behavior.
CHECK
User-defined predicate the DBMS enforces on every insert/update.
Implicit join
Multiple tables in FROM plus the join predicate in WHERE.
Explicit join
JOIN ... ON; clearer, preferred.
Table alias
Short name after the table in FROM; used throughout the query (employee e).
DISTINCT
Removes duplicate rows from the result.
BETWEEN
Inclusive range shorthand: x BETWEEN a AND b.
LIKE
Pattern match with % (any string) and _ (any char).
IN
Membership in a list: col IN (...).
ORDER BY
Sort the output; ASC default or DESC.
