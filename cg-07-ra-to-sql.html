Relational Algebra to SQL Translation

Every RA operator mapped to a concrete SQL pattern, with the Division pattern walked through end-to-end.

Relational algebra is the blueprint; SQL is the building. An architect sketches $\sigma_{P}(R)$ on paper, then a contractor turns it into select * from R where P. Same idea, two dialects. Once you learn the translation table below, you can read an RA expression and write the SQL by rote, the way a chef reads a recipe in French and cooks it without thinking in English first.

Six of the seven RA operators translate in one or two lines. The seventh, division ($\div$), needs a double-nested not exists trick; that is the one pattern worth memorizing cold, because SQL has no native division keyword and the professor tests it almost every year.

Part A: Selection & Projection

PART A

Think of a relation as a spreadsheet. Selection ($\sigma$) is the row filter: "show me only the rows where price > 100." Projection ($\pi$) is the column filter: "show me only the Name and Price columns, and throw away any duplicate rows that appear after hiding the other columns." SQL spells the filters out with where and with the column list in select.

Selection: $\sigma_{P}(R)$ returns every tuple of $R$ that satisfies predicate $P$.

$\sigma$
the selection operator (a row filter)
$P$
a boolean predicate on attributes of $R$, like $\text{price} \gt 100$ or $\text{dept} = \text{'CS'}$
$R$
the input relation (a table)

SQL equivalent:

select * from R where P;

Projection: $\pi_{A_1, A_2, \ldots, A_k}(R)$ returns only the listed columns and removes duplicate rows (RA sets have no duplicates).

$\pi$
the projection operator (a column filter)
$A_i$
attribute names to keep
$R$
the input relation

SQL equivalent:

select distinct A1, A2, ..., Ak from R;

Why distinct matters. SQL is a multiset (bag) language by default. If you write select dept from employee you will get the department name once per employee. RA's $\pi$ is a set operator, so it automatically removes duplicates. Adding distinct is how you bridge the semantic gap. If the exam asks you to write SQL for $\pi_{\text{dept}}(\text{Employee})$ and you forget distinct, the answer is technically wrong.

Example: filter and project

RA expression

$\pi_{\text{name, salary}}(\sigma_{\text{salary} \gt 50000}(\text{Employee}))$

Read it right-to-left

First apply the row filter, then keep only two columns, then drop duplicates.

SQL
select distinct name, salary from employee where salary > 50000;

Inside-out nesting in RA becomes top-to-bottom clauses in SQL.

Part B: Cartesian Product and Join

PART B

Cartesian product ($\times$) is every pairing of two decks of cards laid side by side; if deck R has 4 cards and deck S has 13 cards, you end up with 52 rows. A join ($\bowtie$) is Cartesian product followed by a filter that keeps only the pairings where the shared attribute matches (so a Heart from R pairs only with a Heart from S). SQL does the product with a comma in the from clause and does the filter with where, or spells it out with natural join.

Cartesian product: $R \times S$ pairs every tuple of $R$ with every tuple of $S$; the result has $|R| \cdot |S|$ rows and all attributes of both.

SQL equivalent:

select * from R, S;

Natural join: $R \bowtie S$ is the Cartesian product restricted to tuples that agree on every shared attribute, and duplicate shared columns are folded into one.

SQL (explicit form):

select * from R, S where R.shared = S.shared;

SQL (keyword form):

select * from R natural join S;

Shared-attribute gotcha. If $R$ and $S$ share attributes a and b, the explicit form needs both equalities: where R.a = S.a and R.b = S.b. A missing equality silently degrades the join back to a partial Cartesian product (usually wrong, but valid SQL).

Example: natural join Instructor with Department

RA expression

$\text{Instructor} \bowtie \text{Department}$, sharing attribute dept_name.

SQL (explicit)
select * from instructor as i, department as d where i.dept_name = d.dept_name;
SQL (keyword)
select * from instructor natural join department;

Both produce the same rows; the keyword form is shorter but hides which attribute is the join key.

Part C: Set Operations

PART C

Set operations are Venn-diagram shading. Union ($\cup$) shades both circles; intersection ($\cap$) shades only the overlap; difference ($-$) shades one circle minus the overlap. In SQL these become union, intersect, except, and they require both inputs to have the same columns in the same order (union-compatibility).

Union: $R \cup S$ keeps every tuple in either relation (duplicates removed).

(select ... from R) union (select ... from S);

Difference: $R - S$ keeps every tuple in $R$ that is not in $S$.

(select ... from R) except (select ... from S);

Intersection: $R \cap S$ keeps every tuple in both.

(select ... from R) intersect (select ... from S);

union vs union all. Plain union removes duplicates (matches RA). union all keeps duplicates. Same story for intersect all and except all. If the exam says "relational algebra style," use the plain forms.

Example: customers who have either a loan or an account

RA expression

$\pi_{\text{cust\_id}}(\text{Loan}) \cup \pi_{\text{cust\_id}}(\text{Account})$

SQL
(select distinct cust_id from loan) union (select distinct cust_id from account);

The distinct inside each branch is technically redundant because union already removes duplicates, but it mirrors the RA expression literally.

Part D: Division (the hard one)

PART D

Division ($\div$) answers universal questions: "which students have taken every required course?" Imagine a list of required courses $S$ and a big table $R$ of (student, course) enrollments. Division asks for each student $s$ whether the set of courses $s$ has taken is a superset of $S$; if yes, keep $s$. It is the "for all" quantifier of relational algebra.

SQL has no divide keyword. Instead we simulate the "for all" using the identity: "for every $x$ in $S$, property $P$ holds" is equivalent to "there does not exist an $x$ in $S$ such that $P$ fails." That double negation is what the not exists + except pattern captures.

Division: Let $R$ have attributes $(\alpha, \beta)$ and $S$ have attributes $\beta$. Then $R \div S$ returns the set of $\alpha$-values $r$ such that for every $s \in S$, the tuple $(r, s)$ appears in $R$.

$R$
the big relation, schema $(\alpha, \beta)$
$S$
the divisor, schema $\beta$ (a subset of $R$'s attributes)
$\alpha$
the attributes unique to $R$ (the "result" attributes)
$\beta$
the attributes shared between $R$ and $S$

SQL pattern (A2 Q5): using a correlated subquery with not exists and except.

select r.A from r as onetuple where not exists (   (select * from s)   except   (select r.B, r.C from r where r.A = onetuple.A) );

Read the except as "the set of s-values not yet taken by this student." The outer not exists says "this set is empty," which is exactly "the student has taken every s-value." Double negation, done.

A2 Q5 pattern: students who took every required course

Schema: enrollment(student_id, course_id) and required(course_id). Goal: list every student who has taken every course in required.

Step 1: RA expression

$\pi_{\text{student\_id}}(\text{Enrollment}) \div \text{Required}$

Step 2: restate in English using double negation

"Every student $s$ such that there does not exist a required course $c$ where $(s, c)$ is missing from Enrollment."

Step 3: translate to SQL
select distinct e.student_id from enrollment as e where not exists (   (select course_id from required)   except   (select course_id from enrollment    where student_id = e.student_id) );
Step 4: trace the inner except for one student

Suppose required = {CPSC3660, CPSC2620}. For Alice whose enrollment rows are {(Alice, CPSC3660), (Alice, CPSC2620), (Alice, CPSC2720)}: the right-hand side of except produces {CPSC3660, CPSC2620, CPSC2720}. Then {CPSC3660, CPSC2620} except {CPSC3660, CPSC2620, CPSC2720} = {}. Empty, so not exists is true, so Alice is kept.

Step 5: trace for a student missing a course

For Bob whose rows are {(Bob, CPSC3660)}: the right-hand side produces {CPSC3660}. Then {CPSC3660, CPSC2620} except {CPSC3660} = {CPSC2620}. Non-empty, so not exists is false, so Bob is dropped.

The not exists + except sandwich is the workhorse of every "every/all/for all" question in SQL.

Exam Pattern

A2 Q3-Q5 established this pattern for CPSC 3660, and the final almost always has one division-style question. Memorize the five-line skeleton verbatim and only fill in the attribute names. If you see words like "every," "all," "for each required," reach for this pattern first; do not try to write it as a join or group by with count unless the question explicitly asks for aggregation.

Why This Matters

Every SQL query can be rewritten as an RA expression, and every RA expression can be rewritten as SQL. Knowing the translation both ways is the point of this course; it is the scaffolding under every database engine. The optimizer inside Postgres or Oracle does exactly what you are doing by hand: it reads your SQL, converts it to a relational-algebra tree, shuffles the tree for efficiency, then emits an execution plan. Part A of the final almost always asks for one translation in each direction.

Part F: Translation Checklist

PART F
RA-to-SQL checklist
1. read RA right-to-left: inner operators happen first 2. every $\pi$ gets a distinct 3. every $\sigma$ becomes a where clause 4. $\times$ becomes a comma in from 5. $\bowtie$ becomes natural join or an equality in where 6. $\cup, -, \cap$ become union, except, intersect 7. $\div$ becomes not exists + except (Part D) 8. $\gamma$ becomes group by + aggregate function

Exam Pattern

When the final shows an RA tree and asks for SQL, label the operators and translate them one at a time. Do not try to write the whole query in one pass; you will forget a distinct or a join key. Write the innermost piece first as a subquery, then wrap it.

Glossary

Union-compatible
Two relations have the same number of attributes, in the same order, with compatible data types. Required for $\cup$, $\cap$, $-$.
Correlated subquery
An inner SQL query that references a column from the outer query (like onetuple.A above). Re-evaluated for every outer row.
Double negation trick
Rewriting "for all $x$, $P(x)$" as "there does not exist $x$ such that $\neg P(x)$." The foundation of the division-in-SQL pattern.
Division ($\div$)
RA operator that answers "which $\alpha$-values are paired with every $\beta$-value in the divisor $S$." No direct SQL keyword; simulated with not exists and except.
Natural join ($\bowtie$)
Equi-join on every shared attribute, with duplicate shared columns collapsed.
Theta join
Cartesian product followed by an arbitrary predicate filter; written $R \bowtie_{\theta} S$.
