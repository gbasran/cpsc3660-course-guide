Normalization and BCNF

The craft of slicing a schema so that every fact lives in exactly one place: 1NF through BCNF, lossless-join, dependency preservation.

Normalization is the art of un-mixing your laundry. A bad schema throws socks, shirts, and pants into one pile; you cannot update a shirt's color without risking staining a sock, and deleting the last pair of pants might also delete the knowledge that pants exist as a concept. Normalization separates the pile into three drawers (socks, shirts, pants) so each garment has a home and redundancy disappears.

The normal forms are progressively stricter standards for how clean the separation must be. 1NF is the bare minimum (no nested lists inside a cell). 2NF adds "nothing about a piece of a key should live with the key." 3NF adds "nothing non-key should depend on another non-key." BCNF is the gold standard: every determinant is a superkey. You aim for BCNF and sometimes settle for 3NF when BCNF would cost you a dependency.

Part A: The Normal Forms (1NF, 2NF, 3NF, BCNF)

PART A

Each normal form fixes a specific kind of redundancy. 1NF bans nested structure, 2NF bans key-piece dependencies, 3NF bans transitive dependencies, BCNF bans every non-superkey determinant. The higher you go, the fewer update anomalies you can suffer.

1NF (first normal form). Every attribute is atomic; no cell contains a list, set, or nested record.

Example violation: a Student(id, name, courses) row where courses = {CPSC3660, MATH1410}. Fix: move courses to a separate Enrollment(student_id, course_id) table.

2NF. The schema is in 1NF and no non-prime attribute is partially dependent on a candidate key. "Partial dependency" means it depends on only part of a composite key.

Example violation: Enrollment(student_id, course_id, student_name) with key $\{student\_id, course\_id\}$. The FD $student\_id \to student\_name$ is partial because it uses only half the key. Fix: move student_name to a Student table.

3NF. For every non-trivial FD $\alpha \to \beta$ in $F^+$, at least one of these holds:

  1. $\alpha$ is a superkey, or
  2. every attribute in $\beta - \alpha$ is part of some candidate key (i.e., is "prime").

3NF forbids transitive dependencies on non-prime attributes but allows prime-attribute transitives, which is the escape hatch that sometimes lets 3NF preserve dependencies when BCNF cannot.

BCNF (Boyce-Codd Normal Form). For every non-trivial FD $\alpha \to \beta$ in $F^+$, $\alpha$ is a superkey of $R$.

In plain English: the only way an attribute set can determine another is if it can determine the entire row. No "side channels" of information.

Hierarchy. BCNF $\subseteq$ 3NF $\subseteq$ 2NF $\subseteq$ 1NF. Every BCNF schema is automatically 3NF, 2NF, 1NF. The reverse is not true.

Part B: Candidate Key Enumeration

PART B

A candidate key is a minimal subset of attributes that acts as a unique identifier. To find all of them, you compute closures for progressively larger attribute sets until you find every minimal set whose closure covers the whole schema. It is a treasure hunt: start small, grow until you strike gold (cover $R$), back off any redundant attribute, record the minimal set.

Algorithm: enumerate candidate keys of $(R, F)$
1. for each subset $\alpha \subseteq R$ (increasing size): 2.   compute $\alpha^+$ 3.   if $\alpha^+ = R$ and no proper subset of $\alpha$ is a key: 4.     record $\alpha$ as a candidate key 5. return list of candidate keys

Shortcut. Any attribute that appears on no RHS of any FD must be in every candidate key (nothing can derive it). Any attribute on no LHS and only on a RHS is never in a candidate key by itself. Use these to prune the search.

A4 Q6 setup: $R = (A, B, C, D, E)$, $F = \{A \to BC, CD \to E, B \to D, E \to A\}$

Let us find every candidate key by computing closures.

Check single attributes

$A^+ = \{A, B, C, D, E\}$ (via $A \to BC$, $B \to D$, $CD \to E$). Covers $R$. So $\{A\}$ is a superkey; minimal (only one attribute), so candidate key.

$B^+ = \{B, D\}$. Not a superkey.

$C^+ = \{C\}$. Not a superkey.

$D^+ = \{D\}$. Not a superkey.

$E^+ = \{A, B, C, D, E\}$ (via $E \to A$ then chain). Covers $R$. Candidate key $\{E\}$.

Check pairs (skip pairs containing $A$ or $E$: not minimal)

$BC^+$: start $\{B, C\}$, $B \to D$ adds $D$ ($\{B, C, D\}$), $CD \to E$ adds $E$ ($\{B, C, D, E\}$), $E \to A$ adds $A$ (all). Candidate key $\{BC\}$.

$BD^+ = \{B, D\}$, not a superkey.

$CD^+$: start $\{C, D\}$, $CD \to E$ adds $E$, $E \to A$ adds $A$, $A \to BC$ adds $B$ (all). Candidate key $\{CD\}$.

Larger sets

Anything strictly larger than a candidate key is a superkey but not minimal. Stop.

Candidate keys = $\{A\}, \{BC\}, \{CD\}, \{E\}$.

Part C: BCNF Violation Detection

PART C

Violation test. Scan every non-trivial FD $\alpha \to \beta$ in $F$. For each, compute $\alpha^+$. If $\alpha^+ \neq R$, then $\alpha$ is not a superkey and the FD is a BCNF violation.

A4 Q6: check each FD

$A \to BC$

$A^+ = R$. $A$ is a superkey. OK.

$CD \to E$

$CD^+ = R$. $CD$ is a superkey. OK.

$B \to D$

$B^+ = \{B, D\}$. Not a superkey. VIOLATION.

$E \to A$

$E^+ = R$. $E$ is a superkey. OK.

Exactly one BCNF violation: $B \to D$. We decompose on this FD next.

Part D: BCNF Decomposition Algorithm (A4 Q6 walkthrough)

PART D

When you find a BCNF violation $\alpha \to \beta$, you split the bad relation into two pieces: one piece holds the offending dependency $(\alpha, \beta)$ as its own table so that $\alpha$ becomes a key there, and the other piece holds everything else plus $\alpha$ (so the two halves can still be joined back together by matching on $\alpha$). Then you recursively check each half for further violations. You stop when every fragment is in BCNF.

It is like splitting a noisy apartment building: move the loud tenants to a separate unit next door, keep the shared hallway (the $\alpha$ attribute) so visitors can still find both groups.

Algorithm: decomposeBCNF($R, F$)
1. result := $\{R\}$ 2. while some $R_i \in$ result has a BCNF violation $\alpha \to \beta$ with $\alpha, \beta \subseteq R_i$: 3.   $R_a := \alpha \cup \beta$  (the violating FD lives here) 4.   $R_b := R_i - (\beta - \alpha)$  (everything else, keep $\alpha$ as the link) 5.   replace $R_i$ in result with $R_a, R_b$ 6. return result

Full A4 Q6 decomposition

Starting point: $R = (A, B, C, D, E)$, $F = \{A \to BC, CD \to E, B \to D, E \to A\}$.

Violation: $B \to D$.

Step 1: split on $B \to D$

$R_1 = \{B\} \cup \{D\} = (B, D)$.

$R_2 = R - (\{D\} - \{B\}) = (A, B, C, E)$.

Step 2: check $R_1 = (B, D)$ for BCNF

FDs restricted to $R_1$: only $B \to D$. Compute $B^+$ within the full $F$: $\{B, D\}$, which equals $R_1$. So $B$ is a superkey of $R_1$. BCNF.

Step 3: check $R_2 = (A, B, C, E)$ for BCNF

FDs restricted to $R_2$ (drop anything mentioning $D$): $A \to BC$, $E \to A$. Also $CD \to E$ and $B \to D$ are lost because $D \notin R_2$.

Candidate keys of $R_2$: compute closures under $F$ within $R_2$.

$A^+_{R_2} = \{A, B, C\} \cup \ldots$: we can still use $A \to BC$ (all attributes in $R_2$) so $\{A, B, C\}$, then $E \to A$ does nothing new. Need $E$? $A$ does not determine $E$ in $R_2$'s restricted FD set. But under the full $F$, $A^+ = R$, which includes $E$. Within $R_2$ projection: $A \to B, A \to C$. Chain: does $A \to E$ hold in $F^+$? Yes, via $A \to BC$, $B \to D$, $CD \to E$; but $D$ is outside $R_2$ so this path uses a missing attribute. Still, $F^+$ is a global concept; $A \to E$ is in $F^+$ even though its derivation passes through $D$. So $A^+_{R_2}$ using $F^+$ restricted to $R_2$ attributes is $\{A, B, C, E\}$. Candidate key $\{A\}$.

$E^+_{R_2}$: $E \to A$, $A \to BC$, all in $R_2$. $E^+_{R_2} = \{A, B, C, E\} = R_2$. Candidate key $\{E\}$.

$B^+_{R_2}$: $B$ determines $D$ in full $F$, but $D \notin R_2$. Does $B$ determine anything in $R_2$? Not through any projected FD. $B^+_{R_2} = \{B\}$. Not a key.

Check each FD in $R_2$ against the "$\alpha$ is superkey" rule:

$A \to BC$: $A$ is candidate key of $R_2$. OK.

$E \to A$: $E$ is candidate key of $R_2$. OK.

$R_2$ is in BCNF.

Final decomposition

$\{R_1, R_2\} = \{(B, D), (A, B, C, E)\}$. Both BCNF.

Part E: Lossless Decomposition Test

PART E

Lossless means you can always put Humpty Dumpty back together. Take the original relation, split it into two fragments, natural-join the fragments; if you get back exactly the original tuples (no extras, no missing), the decomposition is lossless. If extra spurious tuples appear, the decomposition is lossy and you have corrupted your data.

Jigsaw puzzle analogy: a lossless split is like cutting a jigsaw along its natural edges; you can reassemble it uniquely. A lossy split is like cutting through the middle of pieces; when you try to reassemble, multiple fits are possible and extra ghost pieces appear.

Lossless-join test. A decomposition of $R$ into $R_1$ and $R_2$ is lossless iff at least one of the following FDs is in $F^+$:

  1. $(R_1 \cap R_2) \to R_1$, or
  2. $(R_1 \cap R_2) \to R_2$.

Equivalently: the shared attributes between the two fragments must form a superkey of at least one of them.

Lossless example (Week 14): Borrow-loan

$R = (\text{customerID}, \text{loan\_number}, \text{amount})$, FDs: $\text{loan\_number} \to \text{amount}$.

Decompose

$R_1 = (\text{customerID}, \text{loan\_number})$, $R_2 = (\text{loan\_number}, \text{amount})$.

Shared attributes

$R_1 \cap R_2 = \{\text{loan\_number}\}$.

Check superkey

Does $\text{loan\_number} \to R_2$? $\text{loan\_number}^+ = \{\text{loan\_number}, \text{amount}\} = R_2$. Yes.

Lossless. Two customers can share a loan; you can still rejoin uniquely.

PART F

Lossy example (Week 14): Employee-Project-Branch

Why it failed

$R_1 \cap R_2 = \{\text{Branch}\}$. But $\text{Branch}$ does not determine $\text{EmployeeID}$ and does not determine $\text{Project}$. Neither $\text{Branch} \to R_1$ nor $\text{Branch} \to R_2$ holds.

Lossy. Broken-mirror analogy: once you shatter along the wrong fault line, you cannot reconstruct the original reflection without ghosts.

BCNF decomposition is always lossless when you split on a violating FD $\alpha \to \beta$ with $R_1 = \alpha \cup \beta$ and $R_2 = R - (\beta - \alpha)$, because then $R_1 \cap R_2 = \alpha$ and $\alpha \to \beta$ (the FD we split on) means $\alpha \to R_1$. Lossless by construction.

Part G: Dependency Preservation Test

PART G

Dependency preservation asks whether every FD that was enforceable on the original schema can still be enforced locally on the fragments. If a BCNF decomposition scatters the LHS and RHS of some FD across two different fragments, checking that FD requires a join, which is expensive and often needs a trigger. The schema is "weaker" than the original in a practical sense.

Think of it like splitting a contract between two lawyers: if every clause is fully contained in one lawyer's half, either lawyer can enforce it independently. If a clause spans both halves, enforcement requires both to meet and compare notes (a join); the clause is "preserved" only in the abstract.

FD restricted to $R_i$. $F_i = \{\alpha \to \beta \in F^+ : \alpha\beta \subseteq R_i\}$. Only FDs whose both sides live entirely inside $R_i$.

Dependency preservation test. Let $F' = F_1 \cup F_2 \cup \ldots \cup F_n$. The decomposition is dependency-preserving iff $(F')^+ = F^+$.

Preserved example (Week 14)

$R = (A, B, C)$, $F = \{A \to B, B \to C, C \to B\}$.

Decomposition

$R_1 = (A, C)$, $R_2 = (B, C)$.

Restrict $F$ to each fragment

$F_1$ on $R_1 = (A, C)$: do any FDs (or derivable FDs) have both sides in $\{A, C\}$? $A \to B$ no. $B \to C$ no. $C \to B$ no. But derivable: $A \to C$ (via $A \to B \to C$). So $F_1 = \{A \to C\}$.

$F_2$ on $R_2 = (B, C)$: $B \to C$ yes. $C \to B$ yes. $F_2 = \{B \to C, C \to B\}$.

Check $(F')^+ = F^+$

$F' = \{A \to C, B \to C, C \to B\}$. Can we derive $A \to B$ from $F'$? $A \to C$ and $C \to B$ chain to give $A \to B$. Yes. So $F'$ implies everything $F$ implies.

Dependency-preserving.

Not-preserved example (Week 14)

$R = (A, B, C)$, $F = \{C \to B, AB \to C\}$.

Decomposition

$R_1 = (A, C)$, $R_2 = (B, C)$.

Restrict

$F_1$ on $R_1 = (A, C)$: $C \to B$ no ($B \notin R_1$). $AB \to C$ no ($B \notin R_1$). Any derivable FD with both sides in $\{A, C\}$? No. $F_1 = \emptyset$.

$F_2$ on $R_2 = (B, C)$: $C \to B$ yes. $AB \to C$ no. $F_2 = \{C \to B\}$.

Check

$F' = \{C \to B\}$. Can we derive $AB \to C$ from $F'$? No; $C$ is never on the RHS of any rule except its own trivial reflexive ones. $(F')^+ \subsetneq F^+$.

Not dependency-preserving. To enforce $AB \to C$ you would need to join $R_1$ and $R_2$.

Part H: Back to A4 Q6 — Is It Dependency-Preserving?

PART H

Check A4 Q6 decomposition for dependency preservation

Decomposition: $R_1 = (B, D)$, $R_2 = (A, B, C, E)$. Original $F = \{A \to BC, CD \to E, B \to D, E \to A\}$.

Restrict to $R_1 = (B, D)$

$B \to D$ has both sides in $R_1$. Keep. $F_1 = \{B \to D\}$.

Restrict to $R_2 = (A, B, C, E)$

$A \to BC$: both sides in $R_2$. Keep.

$CD \to E$: $D \notin R_2$. Drop.

$B \to D$: $D \notin R_2$. Drop.

$E \to A$: both sides in $R_2$. Keep.

$F_2 = \{A \to BC, E \to A\}$.

Compute $F' = F_1 \cup F_2$

$F' = \{B \to D, A \to BC, E \to A\}$.

Is $CD \to E$ in $(F')^+$?

Compute $CD^+$ under $F'$: start $\{C, D\}$. Is any FD LHS a subset? $B \to D$: no. $A \to BC$: no. $E \to A$: no. No FDs fire. $CD^+_{F'} = \{C, D\}$.

Under the original $F$, $CD^+ = R$, so $CD \to E \in F^+$. But $E \notin CD^+_{F'}$, so $CD \to E \notin (F')^+$.

Not dependency-preserving. The FD $CD \to E$ spans both fragments and cannot be enforced locally.

Exam Pattern

A4 Q5-Q6 is the archetype: "given this schema and FDs, find candidate keys, check BCNF, decompose if needed, verify lossless, verify dependency preservation." Expect the final to have this exact shape with different letters. Memorize the four checks: (1) enumerate CKs, (2) find BCNF violations, (3) decompose, (4) classify each piece as BCNF/lossless/dep-preserving. Always state which FD you decomposed on.

Part I: Trade-Off — BCNF vs 3NF

PART I

Sometimes a schema cannot simultaneously be BCNF, lossless, and dependency-preserving. You can always achieve any two, but occasionally the third must be sacrificed. 3NF is the consolation prize: it is weaker than BCNF but always achievable along with lossless and dependency-preserving. The rule of thumb: if a BCNF decomposition would lose dependencies, fall back to 3NF.

Design goal. Aim for:

  1. BCNF (no redundancy from FDs)
  2. Lossless-join (data can be reconstructed)
  3. Dependency-preserving (FDs can be enforced locally)

If all three are possible, take them. If not, settle for 3NF + lossless + dependency-preserving, which is always achievable via the 3NF synthesis algorithm.

3NF synthesis (for reference). Compute the canonical cover $F_c$ of $F$. For each FD $\alpha \to \beta$ in $F_c$, create a fragment $\alpha\beta$. If no fragment contains a candidate key of $R$, add one fragment that is a candidate key. The result is 3NF, lossless, and dependency-preserving by construction.

Part J: Full Normalization Checklist

PART J
Normalization workflow for the exam
1. list $R$'s attributes and all given FDs $F$ 2. enumerate candidate keys by computing closures 3. for each FD $\alpha \to \beta$ in $F$: check if $\alpha$ is a superkey 4.   if not, mark as BCNF violation 5. pick one violation $\alpha \to \beta$; split:     $R_1 := \alpha \cup \beta$     $R_2 := R - (\beta - \alpha)$ 6. recurse on $R_1$ and $R_2$ with restricted FDs 7. verify lossless: $R_1 \cap R_2 \to R_1$ or $\to R_2$ 8. verify dependency preservation: $(\cup F_i)^+ = F^+$? 9. if dep-preservation fails, note it; optionally fall back to 3NF

Why This Matters

Redundancy breeds anomalies. Update anomalies (forgetting to update one copy of a fact leaves the database inconsistent), insertion anomalies (cannot insert a course unless someone enrolls in it), deletion anomalies (deleting the last enrollment in a course also deletes the course itself). BCNF is the strongest protection against all three, achieved by ensuring every determinant is a superkey; every fact lives in exactly one place. The whole point of Chapters 7-8 of the textbook, and likely 30-40% of the final exam, is teaching you to spot and remove these hazards.

Glossary

1NF
Atomic attributes; no nested collections.
2NF
1NF plus no non-prime attribute is partially dependent on a candidate key.
3NF
For every non-trivial $\alpha \to \beta \in F^+$: $\alpha$ is a superkey, or $\beta - \alpha$ is all prime.
BCNF
For every non-trivial $\alpha \to \beta \in F^+$: $\alpha$ is a superkey. The gold standard.
Prime attribute
Any attribute that appears in some candidate key.
Lossless decomposition
$R_1 \bowtie R_2 = R$; no spurious tuples on rejoin. Test: $R_1 \cap R_2$ is a superkey of at least one fragment.
Dependency preservation
Every FD in $F^+$ can be derived from FDs localized to single fragments.
Update anomaly
Redundant copies of a fact fall out of sync when one is updated.
Insertion anomaly
Cannot represent a fact because its required key partner is missing.
Deletion anomaly
Deleting one tuple accidentally removes an unrelated fact.
Canonical cover $F_c$
A minimal equivalent of $F$: no redundant FDs, no redundant attributes on any LHS.
