Q1: Schema vs Instance

Question (2 marks). Define schema and instance in the context of a relational database, and state how they differ.

Model answer

Schema is the logical structure of the database: relations, attributes, domains, constraints; it changes rarely. Instance is the set of tuples currently stored in those relations; it changes with every INSERT, UPDATE, or DELETE. The schema is fixed; the instance is the current snapshot.

Q2: Super/Candidate/Primary Key Distinction

Question (2 marks). Define super key, candidate key, and primary key. Give an example of each using the relation $employee(empid, name, office, age)$.

Model answer

A super key is any set of attributes that uniquely identifies a tuple (e.g. $\{empid, name\}$). A candidate key is a minimal super key, with no redundant attributes (e.g. $\{empid\}$). A primary key is the candidate key chosen as the official row identifier (here, $empid$); it must be NOT NULL and unique.

Mirrors A1 Q3; Midterm Q2

Q3: DML vs DDL

Question (2 marks). Define DML and DDL. Give one SQL example of each.

Model answer

DDL (Data Definition Language) defines and alters schema structure; example: create table employee(...). DML (Data Manipulation Language) queries and modifies the data inside those structures; example: insert into employee values(...) or select * from employee.

Mirrors A1 Q1; Midterm Q1

Q4: Data Dictionary Purpose

Question (2 marks). What is the data dictionary, and why does the DBMS need it?

Model answer

The data dictionary (system catalog) is a set of tables describing every relation, attribute, key, constraint, view, and privilege in the database. The DBMS uses it to validate queries (does this table exist? is this column the right type?), enforce constraints, and feed statistics to the optimizer.

Mirrors A1 Q2; Midterm-style short answer

Q5: Primary Key Uniqueness

Question (2 marks). If $K$ is the primary key of relation $R$, and two tuples $t_1, t_2 \in R$ satisfy $t_1[K] = t_2[K]$, what can you conclude about the other attributes?

Model answer

Since $K$ uniquely identifies each tuple, $t_1[K] = t_2[K]$ forces $t_1 = t_2$; they are the same tuple. Therefore $t_1[A] = t_2[A]$ for every attribute $A$ of $R$, not just for $K$. The key's uniqueness propagates to all attributes.

Mirrors Midterm Q3

Q6: Set Intersection via Basic RA Operations

Question (2 marks). Show how $r \cap s$ can be expressed using only the basic RA operators (selection, projection, union, set-difference, Cartesian product, rename).

Model answer

Intersection is not a basic operator but reduces to two set-differences: $r \cap s = r - (r - s)$. The inner $(r - s)$ removes tuples of $r$ not in $s$; subtracting again from $r$ leaves only the tuples that are in both.

Mirrors A1 Q4; Midterm Q6

Q7: Cascading Delete Rationale

Question (2 marks). What is ON DELETE CASCADE? When should a designer use it rather than RESTRICT?

Model answer

ON DELETE CASCADE means that deleting a parent row automatically deletes every child row referencing it via the foreign key. Use CASCADE when child rows have no meaning without the parent (e.g. borrow rows when the employee is gone); use RESTRICT when child rows are historical records that should block casual deletion.

Q8: Strong vs Weak Entity

Question (2 marks). Define strong entity and weak entity. Give one example of each.

Model answer

A strong entity has its own primary key and exists independently (e.g. $employee$). A weak entity cannot be identified by its own attributes alone and requires the key of an owner entity plus a local discriminator (e.g. $dependent$ of an employee, identified by $(empid, name)$).

Q9: Multi-valued Attribute Decomposition

Question (2 marks). An $employee$ entity has a multi-valued attribute $phone$. How is this mapped to relations?

Model answer

Create a separate relation $employee\_phone(empid, phone)$ where $empid$ is a foreign key referencing $employee$ and $(empid, phone)$ is the primary key. Each phone value becomes one row; the original $employee$ table holds the single-valued attributes only.

Mirrors A3 Q4; A3 Q8

Q10: Composite Attribute Decomposition

Question (2 marks). An $employee$ entity has a composite attribute $address$ with sub-parts $street$, $city$, $postal\_code$. How is this mapped to a relation?

Model answer

Flatten the composite attribute: each leaf sub-attribute becomes its own column in the $employee$ relation. The mapped schema is $employee(empid, name, street, city, postal\_code)$; the grouping $address$ is conceptual only, not stored as a single column.

Mirrors A3 Q4

Q11: Weak Entity Rationale

Question (2 marks). Why do we model some entities as weak instead of just adding a unique identifier?

Model answer

A weak entity exists only in the context of its owner and has no natural standalone key (e.g. a $room$ within a $building$; room "101" is only meaningful relative to its building). Modelling it weak preserves the identifying dependency, enforces mandatory total participation, and avoids inventing artificial IDs.

Mirrors A3 Q5; Midterm-style short answer

Q12: Generalization / ISA Purpose

Question (2 marks). What does an ISA (generalization) relationship in an ER diagram represent, and why is it useful?

Model answer

An ISA relationship groups two or more lower-level entity types (subtypes) under a common higher-level type (supertype); subtypes inherit the supertype's attributes and add their own. It reduces redundancy when subtypes share attributes (e.g. $book$ and $music$ both ISA $item$, sharing $isbn, title, price$).

Mirrors A3 Q8 (online bookstore with music)

Q13: Functional Dependency Definition

Question (2 marks). Define a functional dependency $\alpha \to \beta$ on a relation $R$.

Model answer

$\alpha \to \beta$ holds on $R$ if, for any two tuples $t_1, t_2 \in R$ with $t_1[\alpha] = t_2[\alpha]$, it must also be that $t_1[\beta] = t_2[\beta]$. Informally: knowing the values of $\alpha$ uniquely determines the values of $\beta$.

Mirrors A4 Q1; New territory for the final

Q14: BCNF Definition

Question (2 marks). State the definition of Boyce-Codd Normal Form (BCNF). Why do we aim for it?

Model answer

A relation $R$ is in BCNF if for every non-trivial FD $\alpha \to \beta$ that holds on $R$, $\alpha$ is a super key of $R$. We aim for BCNF to eliminate redundancy and the update/insertion/deletion anomalies that arise when non-key attributes determine other non-key attributes.

Mirrors A4 Q2; A4 Q6

Q15: Lossless Decomposition Criterion

Question (2 marks). A decomposition of $R$ into $R_1, R_2$ is lossless-join under FD set $F$ when what condition holds?

Model answer

The decomposition is lossless when $R_1 \cap R_2 \to R_1$ or $R_1 \cap R_2 \to R_2$ is implied by $F$; that is, the shared attributes functionally determine one of the fragments. Equivalently, $R_1 \bowtie R_2 = R$ for every legal instance, so no spurious tuples are created by the natural join.

Glossary

Schema
Logical structure of the database; relations, attributes, domains, constraints.
Instance
Current set of tuples stored in those relations.
DDL / DML
Data definition vs data manipulation languages; schema changes vs data changes.
Super key / candidate key / primary key
Unique attribute set / minimal unique set / chosen unique set.
Data dictionary
Metadata tables describing the schema; used by DBMS for validation and optimization.
ON DELETE CASCADE
FK option that deletes child rows when the parent row is deleted.
Strong / weak entity
Entity with / without its own standalone primary key.
Multi-valued attribute
Attribute that takes a set of values; mapped to its own relation.
Composite attribute
Attribute that has sub-parts; flattened into its leaf attributes.
Generalization (ISA)
Higher-level entity type whose subtypes inherit attributes.
Functional dependency
$\alpha \to \beta$ means $\alpha$ uniquely determines $\beta$.
BCNF
Every non-trivial FD $\alpha \to \beta$ has $\alpha$ as a super key.
Lossless decomposition
Common attributes determine at least one fragment; natural-join back produces no spurious tuples.
