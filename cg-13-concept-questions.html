Q1: Schema vs Instance

Question (2 marks). Define schema and instance in the context of a relational database, and state how they differ.
Step 1 Two terms, one contrast; define each, then name the axis they differ on.
Step 2 Anchor words: schema = structure; instance = data at this moment.
Step 3 Finish with "schema static, instance changes with DML" to clinch the 2nd mark.
schema = structure (tables, cols, FKs)
instance = rows right now
→ static vs snapshot
Q1: Schema = logical structure (relations, attributes, domains, constraints); rarely changes.
Instance = set of tuples currently stored; changes with every insert/update/delete. // schema fixed, instance is a snapshot

Q2: Super/Candidate/Primary Key Distinction

Question (2 marks). Define super key, candidate key, and primary key. Give an example of each using the relation $employee(empid, name, office, age)$.
Step 1 Three terms, nested: super key $\supseteq$ candidate key $\supseteq$ primary key.
Step 2 Key word for candidate: "minimal". Key word for primary: "chosen".
Step 3 Pull one example each from the given employee schema.
SK: unique → {empid, name}
CK: minimal SK → {empid}
PK: chosen CK → empid
→ SK $\supseteq$ CK $\supseteq$ PK
Q2: Super key = attr set that uniquely IDs a tuple; e.g. {empid, name}.
Candidate key = minimal super key (no proper subset is a SK); e.g. {empid}.
Primary key = chosen CK, NOT NULL + unique; here empid. // can have >1 CK

Q3: DML vs DDL

Question (2 marks). Define DML and DDL. Give one SQL example of each.
Step 1 Two acronyms, two examples. DDL = structure; DML = data.
Step 2 Pick iconic examples: create table for DDL, select/insert for DML.
DDL: define structure → create table
DML: modify data → select / insert / update
→ structure vs content
Q3: DDL (Data Definition Language) defines/alters schema; e.g. create table employee(...).
DML (Data Manipulation Language) queries/modifies data; e.g. select * from employee.

Q4: Data Dictionary Purpose

Question (2 marks). What is the data dictionary, and why does the DBMS need it?
Step 1 Define what it IS (metadata store), then say WHY the DBMS reads it.
Step 2 Three uses to list: validate, enforce, optimize.
= system catalog (tables about tables)
DBMS uses it to: validate / enforce / optimize
→ metadata lookup
Q4: Data dictionary (system catalog) = tables describing every relation, attribute, key, constraint, view, privilege.
DBMS uses it to validate queries, enforce constraints, and feed statistics to the optimizer.

Q5: Primary Key Uniqueness

Question (2 marks). If $K$ is the primary key of relation $R$, and two tuples $t_1, t_2 \in R$ satisfy $t_1[K] = t_2[K]$, what can you conclude about the other attributes?
Step 1 PK forces uniqueness; matching PK values means the rows ARE the same row.
Step 2 Conclusion extends to every attribute, not just $K$.
$t_1[K] = t_2[K]$ + K is PK
→ $t_1 = t_2$ (same tuple)
→ $t_1[A] = t_2[A]\ \forall A \in R$
Q5: Since $K$ is the PK, $t_1[K] = t_2[K]$ forces $t_1 = t_2$; they are the same tuple.
Therefore $t_1[A] = t_2[A]$ for every attribute $A$ of $R$. // key uniqueness propagates

Q6: Set Intersection via Basic RA Operations

Question (2 marks). Show how $r \cap s$ can be expressed using only the basic RA operators (selection, projection, union, set-difference, Cartesian product, rename).
Step 1 Intersection isn't basic; must use only the 6 allowed ops.
Step 2 Classic trick: double set-difference strips out non-shared tuples twice.
Step 3 Write the identity, then one sentence of intuition.
$r - s$ = in r, not in s
$r - (r - s)$ = in r, and in s
→ $r \cap s = r - (r - s)$
Q6: $r \cap s = \;$$r - (r - s)$.
Inner $(r - s)$ removes tuples of $r$ not in $s$; subtracting again leaves only tuples in both.

Q7: Cascading Delete Rationale

Question (2 marks). What is ON DELETE CASCADE? When should a designer use it rather than RESTRICT?
Step 1 Define mechanism: delete parent → delete children automatically.
Step 2 Contrast with RESTRICT by asking: can the child row mean anything without the parent?
CASCADE: parent del → child del auto
RESTRICT: blocks del if children exist
→ use CASCADE if child worthless w/o parent
Q7: on delete cascade: deleting a parent row auto-deletes every child referencing it via FK.
Use CASCADE when child has no meaning w/o parent (e.g. borrow without employee); RESTRICT for historical records that shouldn't disappear silently.

Q8: Strong vs Weak Entity

Question (2 marks). Define strong entity and weak entity. Give one example of each.
Step 1 Key question: can the entity be identified on its own?
Step 2 Strong = yes, own PK; weak = no, needs owner's key + discriminator.
strong: own PK → employee
weak: owner's PK + discriminator → dependent(empid, name)
→ identifying relationship
Q8: Strong entity: own PK, exists independently; e.g. employee(empid, ...).
Weak entity: cannot be identified alone; needs owner's key + local discriminator; e.g. dependent(empid, name).

Q9: Multi-valued Attribute Decomposition

Question (2 marks). An $employee$ entity has a multi-valued attribute $phone$. How is this mapped to relations?
Step 1 MV attr cannot sit in the base table; 1NF violation.
Step 2 Build a side table with (owner_FK, value) and a composite PK.
employee_phone(empid, phone)
empid → FK to employee
PK = (empid, phone)
→ 1 row per phone
Q9: Create a new relation employee_phone(empid, phone); empid is a FK to employee; PK = (empid, phone).
Each phone value = one row. The original employee table keeps only the single-valued attributes.

Q10: Composite Attribute Decomposition

Question (2 marks). An $employee$ entity has a composite attribute $address$ with sub-parts $street$, $city$, $postal\_code$. How is this mapped to a relation?
Step 1 Composite $\neq$ multi-valued; no side table needed.
Step 2 Flatten: each leaf sub-attr becomes its own column in the base table.
address = (street, city, postal)
address as 1 col
flatten → 3 cols in employee
→ grouping is conceptual only
Q10: Flatten each leaf into its own column:
employee(empid, name, street, city, postal_code). // grouping "address" is conceptual only, not stored

Q11: Weak Entity Rationale

Question (2 marks). Why do we model some entities as weak instead of just adding a unique identifier?
Step 1 Answer the "why not artificial ID?" question; identifying dependency matters.
Step 2 List the 3 benefits: preserves dependency, total participation, no invented IDs.
weak = no standalone meaning
e.g. room(101) only means something w/ building
→ preserves identifying dependency + total participation
Q11: A weak entity exists only via its owner and has no natural standalone key (e.g. room "101" needs its building).
Modelling weak preserves the identifying dependency, forces total participation, avoids inventing artificial IDs.

Q12: Generalization / ISA Purpose

Question (2 marks). What does an ISA (generalization) relationship in an ER diagram represent, and why is it useful?
Step 1 Define: subtypes inherit from a supertype; OOP-style.
Step 2 Explain benefit: removes redundancy when subtypes share attributes.
subtype ISA supertype → inherits attrs
e.g. book ISA item, music ISA item
shared: isbn, title, price
→ reduces attribute redundancy
Q12: ISA: subtypes inherit a supertype's attributes and add their own.
Reduces redundancy when subtypes share attrs; e.g. book, music both ISA item (sharing isbn, title, price).

Q13: Functional Dependency Definition

Question (2 marks). Define a functional dependency $\alpha \to \beta$ on a relation $R$.
Step 1 Formal: two-tuple quantifier; matching $\alpha$ forces matching $\beta$.
Step 2 Informal: knowing $\alpha$ uniquely determines $\beta$.
$\forall\, t_1, t_2 \in R:$
$t_1[\alpha] = t_2[\alpha] \Rightarrow t_1[\beta] = t_2[\beta]$
→ $\alpha$ determines $\beta$
Q13: Formal: $\alpha \to \beta$ on $R$ holds iff $\forall\, t_1, t_2 \in R$: $t_1[\alpha] = t_2[\alpha] \Rightarrow t_1[\beta] = t_2[\beta]$.
Informal: knowing $\alpha$ uniquely determines $\beta$.

Q14: BCNF Definition

Question (2 marks). State the definition of Boyce-Codd Normal Form (BCNF). Why do we aim for it?
Step 1 Formal: every non-trivial FD has a super key on its LHS.
Step 2 Why: removes redundancy and update/insert/delete anomalies.
$\forall$ non-trivial FD $\alpha \to \beta$ on $R$:
$\alpha$ is a super key
→ goal: no anomalies
Q14: $R$ is in BCNF iff for every non-trivial FD $\alpha \to \beta$ on $R$, $\alpha$ is a super key of $R$.
Goal: eliminate redundancy and the update / insert / delete anomalies when non-key attrs determine other non-key attrs.

Mirrors A4 Q2; A4 Q6

Q15: Lossless Decomposition Criterion

Question (2 marks). A decomposition of $R$ into $R_1, R_2$ is lossless-join under FD set $F$ when what condition holds?
Step 1 Criterion: common attrs functionally determine at least one fragment.
Step 2 Equivalent phrasing: natural join back produces exactly $R$, no spurious tuples.
common = $R_1 \cap R_2$
need: common $\to R_1$ OR common $\to R_2$ (from $F^+$)
→ $R_1 \bowtie R_2 = R$
Q15: Lossless iff $F \models (R_1 \cap R_2 \to R_1)$ or $F \models (R_1 \cap R_2 \to R_2)$; common attrs FD a fragment.
Equivalent: $R_1 \bowtie R_2 = R$ for every legal instance. // no spurious tuples from the join

Glossary

Schema
Logical structure of the database; relations, attributes, domains, constraints.
Instance
Current set of tuples stored in those relations.
DDL / DML
Data definition vs data manipulation languages; schema changes vs data changes.
Super key / candidate key / primary key
Unique attribute set / minimal unique set / chosen unique set.
Data dictionary
Metadata tables describing the schema; used by DBMS for validation and optimization.
ON DELETE CASCADE
FK option that deletes child rows when the parent row is deleted.
Strong / weak entity
Entity with / without its own standalone primary key.
Multi-valued attribute
Attribute that takes a set of values; mapped to its own relation.
Composite attribute
Attribute that has sub-parts; flattened into its leaf attributes.
Generalization (ISA)
Higher-level entity type whose subtypes inherit attributes.
Functional dependency
$\alpha \to \beta$ means $\alpha$ uniquely determines $\beta$.
BCNF
Every non-trivial FD $\alpha \to \beta$ has $\alpha$ as a super key.
Lossless decomposition
Common attributes determine at least one fragment; natural-join back produces no spurious tuples.
